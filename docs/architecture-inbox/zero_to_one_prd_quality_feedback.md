# 0-1 普通业务 PRD 质量反馈

- 日期：2026-04-29
- 状态：架构收件箱候选，不是长期规则
- 范围：0-1 普通业务 PRD 的生成质量、结构稳定性、原型图层和回归检查
- 规则：本文件只沉淀已暴露的问题和改进候选，不直接修改 skill、harness、workflow、registry、PRD 模板、pipeline 或长期记忆。

## 处理原则

- 单个测试样例只能作为证据来源，不能直接变成长期规则。
- 每条反馈必须拆成项目证据、通用建议、待拍板动作三层。
- 只有你明确批准后，条目才可以进入模板、prompt、回归测试、real-output eval 或长期规则。
- 不把项目名称写进规则标题，不用测试项目名称绑定长期治理沉淀。

## 反馈 1：测试目的不能误写成产品业务目标

项目证据：

- 生成链路曾把“检查 PRD 结构是否被改坏”这类测试目的，当成产品自身要解决的业务问题。
- 这说明输入解析没有先区分“文档/测试目的”和“产品业务目标”。

通用建议：

- 0-1 PRD intake 应先分离输入里的三类内容：产品事实、文档任务、测试/评审目的。
- PRD 正文只能承接产品事实和明确业务目标；测试目的应进入评审备注或生成记录，不进入产品定位、目标、范围或风险章节。

待拍板动作：

- [ ] 是否把“测试/文档目的与产品业务目标分离”加入 prompt 构建规则。
- [ ] 是否增加 regression 样例，验证测试说明不会进入 PRD 业务目标。
- [ ] 是否把该项纳入 real-output eval，用真实输出持续检查。

## 反馈 2：普通业务项目不能混入非必要 AI 模型选型

项目证据：

- 非 AI 的普通业务 PRD 曾默认出现 AI 模型选型内容。
- 这会让 PRD 读者误以为项目必须包含 AI 架构或模型接入。

通用建议：

- AI 模型选型必须按项目判断：项目涉及 AI 能力、模型调用、智能生成、智能审核、RAG、推荐或自动决策时才出现。
- 非 AI 项目只写普通产品能力、业务流程、页面、权限、指标和风险，不默认生成 AI 模型选型。

待拍板动作：

- [ ] 当前已具备非 AI 不默认输出 AI 模型选型的检查，是否将该规则继续保持为稳定回归要求。
- [ ] 是否增加更多普通业务样例，覆盖出行、交易、内容、工具类项目。

## 反馈 3：PRD 原型图层必须贴合项目页面

项目证据：

- 生成过的可视化图和原型内容曾停留在通用占位，没有真正反映项目用户、页面模块、状态和核心路径。
- 这会导致后续 UI 设计和 Codex 开发文档缺少可靠输入。

通用建议：

- PRD 原型图层至少应包含页面说明、页面跳转关系、页面级低保真原型图或页面级原型说明。
- 原型图层必须使用本项目的角色、页面、状态、权限和异常路径，不能复用无关项目的通用结构。
- PNG、HTML、高保真 UI 仍属于后续阶段，必须在你确认原型方向后再进入。

待拍板动作：

- [ ] 是否把“项目页面贴合度”纳入 PRD 质量检查。
- [ ] 是否为原型图层增加 real-output eval 样例，检查是否存在通用占位。
- [ ] 是否把页面说明、页面跳转关系、原型图层作为 PRD 必备结构继续保持。

## 反馈 4：页面说明、跳转关系、原型图层必须放在对应章节

项目证据：

- 曾出现辅助理解图集中堆在一个“PRD 可视化层”里的问题，导致读者需要来回跳读。
- 原型图层如果脱离对应页面章节，会削弱它对 UI 设计和开发文档的承接价值。

通用建议：

- 产品总览图应靠近摘要或产品概览。
- 流程图、状态流转、权限矩阵、风险闭环应放在对应业务章节。
- 页面说明、页面跳转关系、原型图层应放在页面和原型相关章节。
- 不再默认集中输出“PRD 可视化层”。

待拍板动作：

- [ ] 是否继续把“图表放对应章节”作为 PRD 输出稳定规则。
- [ ] 是否保留“不得重新引入集中式 PRD 可视化层”的 regression 断言。

## 反馈 5：fallback 内容不能串入无关项目

项目证据：

- 普通业务 PRD 曾混入导出、Excel、异步导出等历史项目场景。
- 这说明 fallback 内容存在领域污染风险。

通用建议：

- 通用 fallback 只能生成结构性占位，不能写入具体行业动作。
- 业务动作、状态、页面和异常路径必须来自当前项目输入或领域识别结果。
- 当信息不足时，应显式标注“待业务确认”，不能用历史项目内容补齐。

待拍板动作：

- [ ] 是否增加非导出类项目回归样例，检查导出/Excel/异步任务污染。
- [ ] 是否要求 fallback 输出带来源标记，区分“输入事实”和“待确认假设”。

## 反馈 6：0-1 PRD 必须显式暴露未决问题和边界

项目证据：

- 当输入信息不足时，生成内容容易把不确定项写成确定结论。
- 权限边界、范围边界、指标口径和上线风险如果没有显式评审区，后续开发阶段容易遗漏。

通用建议：

- 0-1 PRD 必须把未决问题、权限边界、范围边界、指标口径、上线风险压到显式评审区。
- 对高风险或信息不足的部分，应标注“待确认”，不能写成已确认方案。
- 用户故事和 Codex 开发文档应承接这些边界，不能绕过 PRD 的待确认项。

待拍板动作：

- [ ] 是否把“未决问题显式评审区”加入 PRD 模板强化项。
- [ ] 是否把权限边界和范围边界纳入 PRD regression / real-output eval。
