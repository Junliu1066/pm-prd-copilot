# 架构自我管理与最小可行治理机制

- 日期：2026-05-03
- 状态：proposal，不是 stable policy。
- 范围：治理架构的上位运行机制，不新增 skill、harness、workflow、plugin、automation。
- 主线任务：把用户确认的“架构是核心、工具服从架构、系统应自我流转、稳定期必须最小可行、用户只做高风险拍板”的思想整理成可审核方案。

## 1. 目标

建立一套长期稳定、可监督、可迭代、自我收敛的架构系统。

这套系统的目标不是堆更多工具，而是让架构能够在收到项目、demo、错误、反馈、AI 情报或用户新方向后，自动完成：

```text
识别类型
-> 拆解可行性
-> 匹配流程
-> 分配职责
-> 执行低风险动作
-> 检查
-> closeout
-> 反哺 architecture inbox
-> 判断 stable / candidate / archive / delete-after-30-days
-> 上报需要用户拍板的事项
```

## 2. 最高原则

| 原则 | 含义 |
|---|---|
| 架构是核心资产 | skill、harness、plugin、automation 都只是工具层，不能反过来绑架架构。 |
| 用户定方向 | 用户负责战略方向、长期取舍和高风险拍板，不负责日常细节优化。 |
| 架构自我管理 | 架构中枢要主动分流、拆解、检查、复盘和上报，而不是等用户逐条提醒。 |
| 探索期可重，稳定期必须轻 | 前期允许重产出暴露问题；进入 stable 后必须最小可行。 |
| 每个 token 都要有价值 | 每个文件、检查、规则、skill、harness 都要证明能降低风险、提升稳定性或提高效率。 |
| 非必要不新增工具 | 能用文档、模板、脚本参数、现有检查解决，就不新增 skill / harness。 |
| 经验先候选 | 单个项目经验不能直接长期化，必须先进入 architecture inbox。 |
| 高风险必须审批 | stable 转正、长期记忆、删除、归档、push / PR、模型成本、外部发布必须用户批准。 |

## 3. 权责结构

用户不是日常优化者，而是战略拍板人。

| 角色 | 职责 |
|---|---|
| 用户 | 定大方向、定取舍、批准 stable 规则、长期记忆、删除、归档、push / PR、模型和成本变化。 |
| 架构中枢 | 判断主线、拆任务、分流、维护看板、避免任务漂移、汇总拍板事项。 |
| 执行部门 | 按职责生成产物、检查、报告、closeout 和反哺候选。 |
| 监督层 | 跑检查、做两轮自检、记录停滞线程、输出中文汇报。 |

## 4. 部门制分工

| 部门 | 职责 | 典型产物 |
|---|---|---|
| 产品部门 | PRD、范围、页面说明、页面跳转关系、原型图层。 | PRD、页面说明、原型说明、验收标准。 |
| 开发部门 | Codex 开发文档、任务包、技术边界、验证命令。 | Codex 开发文档、任务包、技术拆解。 |
| 质量部门 | regression、harness、real-output eval、两轮自检。 | 检查报告、回归结果、质量结论。 |
| 治理部门 | workflow、action、artifact、steward、审批门禁。 | 合同检查、治理报告、审批清单。 |
| 学习部门 | 用户反馈、错误报告、项目 closeout、架构反哺。 | architecture inbox、错误报告、反哺候选。 |
| 归档部门 | raw、zip、缓存、项目产物、30 天删除候选。 | cleanup plan、archive candidate、delete-after-30-days 清单。 |
| 情报部门 | AI / 模型 / 外部变化收集和影响分析。 | AI 情报日报、影响建议、模型更新候选。 |
| 瘦身部门 | 检查 skill、harness、plugin、文档是否还必要。 | pruning review、归档候选、删除候选。 |

## 5. 输入分流机制

任何新输入先分类，再决定是否影响当前主线。

| 输入 | 默认分流 |
|---|---|
| 新项目 / demo | 项目生命周期：可行性 -> PRD / prototype / dev doc -> closeout。 |
| 用户纠正 | 判断一次性修复、项目偏好、长期候选或 stable 变更。 |
| 错误 / 漏洞 | error report -> 根因分析 -> 修复候选 -> 回归检查。 |
| AI 情报 | ai-intel -> 影响分析 -> architecture inbox，不自动改决策文档。 |
| 线程停滞 | 5 天停滞报告 -> 用户确认是否继续、归档或清理。 |
| 新工具想法 | candidate sandbox -> 多项目验证 -> promotion gate。 |

## 6. 架构优化迭代流程

```text
收集
-> 分类
-> 判断是否影响当前主线
-> 拆成最小动作
-> 排优先级
-> 执行
-> 两轮检查
-> closeout / 复盘
-> 进入 architecture inbox
-> 判断 stable / candidate / archive / delete-after-30-days
-> 上报需要用户拍板的事项
```

### 6.1 收集

来源包括用户新方向、项目产物、错误、审查发现、AI 情报、模型更新、线程停滞和 closeout。

### 6.2 分类

先判断输入属于：

- 当前主线。
- 支线补充。
- 长期候选。
- 项目特例。
- 临时材料。

### 6.3 拆小块

每个任务必须写清：

- 为什么做。
- 达到什么效果。
- 下一步最小动作。
- 需要用户拍板什么。
- 不允许做什么。

### 6.4 排优先级

| 优先级 | 定义 |
|---|---|
| P0 | 影响稳定核心或会导致遗漏复发。 |
| P1 | 影响当前治理收口或项目生命周期。 |
| P2 | 提升效率和使用体验。 |
| P3 | 候选想法或低证据材料。 |

### 6.5 执行

默认只推进一个主线任务。支线只有在能降低主线不确定性时才允许做最小检查。

### 6.6 检查

默认两轮：

1. 检查刚改的文件、格式、语法、回归。
2. 检查遗漏、范围、是否新增不必要工具、是否需要更新相关记录。

### 6.7 反哺

有价值的经验进入 architecture inbox 或 proposal，不能直接写入 stable。

## 7. 上下文恢复机制

为防止长线程稀释上下文，使用 `docs/thread_registry.md` 作为恢复入口。

必须维护：

- 当前主线。
- 为什么做。
- 当前做到哪。
- 下一步最小动作。
- 当前不做什么。
- 最近检查结果。
- 需要用户拍板什么。
- 治理任务看板。

WIP 限制：

- 同一时间只允许 1 个 `doing` 主线。
- 最多 1 个 `next` 候选。
- 其它任务必须是 `parked`、`waiting_user` 或 `done`。

## 8. 例外审批制

成熟后的治理不应每一步都让用户把关。默认应由架构中枢按授权矩阵推进，只有出现例外、高风险或越权时才上报用户。

运行原则：

```text
默认推进
-> 按风险分级
-> L1 自动处理
-> L2 汇总成批量审批包
-> L3 单独上报用户
-> 执行后记录账本和检查结果
```

这样用户只负责方向、边界和最终裁决，不负责过程管理。

### 8.1 授权矩阵

| 等级 | 处理方式 | 范围 | 用户介入 |
|---|---|---|---|
| L1 自动处理 | 架构中枢可直接执行，执行后汇报 | 只读检查、临时目录验证、中文报告、台账更新、proposal 分类、check-only、低风险一致性修复 | 不需要逐项确认 |
| L2 批量审批 | 架构中枢先准备方案、清单和验证结果，再集中给用户拍板 | staging 清单、commit 审查、candidate 样例、项目 closeout 审查、proposal 决策记录提交、可回滚的稳定检查补丁 | 用户集中批准 |
| L3 单独审批 | 必须单独上报，未经批准不能执行 | 删除、归档、push / PR、长期记忆、stable policy、新增或稳定 skill / harness、模型供应商 / 成本变化、外部发布、真实项目数据迁移 | 用户单独批准 |

### 8.2 L1 自动处理边界

L1 的目标是减少用户低价值干预。允许自动做：

- 读取和分类现有文件。
- 生成中文 proposal、审查报告、台账更新。
- 在 `/private/tmp` 做临时验证。
- 跑 `git diff --check`、regression、harness check-only。
- 更新非 stable 的 proposal / thread registry，使上下文不丢。
- 给下一步形成建议，不执行高风险动作。

L1 禁止做：

- 不 `git add`、commit、push、PR。
- 不删除、恢复、移动、归档。
- 不写长期记忆。
- 不把 candidate 转 stable。
- 不新增 skill、harness、workflow、plugin、automation。
- 不改模型供应商、模型等级或成本策略。

### 8.3 L2 批量审批边界

L2 的目标是把可回滚但有影响的动作做成批量审批包，减少用户逐步确认。

L2 审批包必须包含：

- 为什么做。
- 涉及文件清单。
- 不包含哪些文件。
- 风险和回滚方式。
- 检查结果。
- 我的建议。
- 用户批准后要执行的精确命令或精确动作。

L2 可以合并问一次，但不能默认执行。典型例子：

- C11 这种 candidate 样例 + regression 修复的 staging / commit 包。
- proposal 决策记录候选的小批次提交。
- 项目 closeout 审查材料是否提交。
- 可回滚的候选检查器或样例库补丁。

### 8.4 L3 例外审批边界

L3 是永久人工审批区。触发 L3 时，架构必须停下来单独问用户。

L3 包括：

- 删除或硬删除。
- 归档项目或清空缓存。
- 长期记忆写入。
- stable policy 生效。
- 新增或稳定 skill / harness。
- push / PR / 外部发布。
- 模型供应商、模型档位或成本策略变化。
- 涉及敏感数据、真实客户数据、外部分发包的不可逆动作。

### 8.5 授权校准

授权不是一次性放开。试运行期间每次汇报都记录：

- 本次 L1 自动处理了什么。
- 本次 L2 建议你批量批准什么。
- 本次 L3 是否触发。
- 我的判断和你的最终判断是否一致。
- 如果不一致，下一次如何收窄授权。

目标是让架构逐步贴近用户判断，而不是靠用户持续手动管理。

## 9. Promotion Gate

候选能力、项目经验或新规则转 stable 前必须满足：

1. 有明确问题背景。
2. 不只是单项目特例。
3. 已比较替代方案。
4. 已证明现有文档、模板、脚本参数或现有检查不足。
5. 有多项目证据或高风险事故证据。
6. 通过 regression / harness / targeted checks。
7. 用户明确批准。

未满足时，保持 candidate。

## 10. Pruning Gate

以下场景必须触发瘦身审查：

- 项目 closeout。
- 版本更新。
- 模型更新。
- 模型供应商 / SDK / API / 价格变化。
- 线程 5 天停滞。
- 归档后 30 天。
- 工作区变更堆积。

瘦身判断：

- 是否重复。
- 是否项目特例。
- 是否还能降低真实风险。
- 是否维护成本过高。
- 是否可由文档、模板或现有检查替代。
- 是否应归档或列入 30 天后删除候选。

## 11. 自动化放权原则

当前不创建新自动化。后续如果要自动化，按阶段放权：

| 阶段 | 范围 |
|---|---|
| 人工监督 | 我生成报告和建议，用户集中审核。 |
| 半自动 | 低风险分类、检查、停滞提醒可自动生成草案。 |
| 低风险自动 | 判断一致率稳定后，允许自动更新台账和候选清单。 |
| 永久人工审批 | stable 转正、长期记忆、删除、归档、push / PR、外部发布、模型成本变化。 |

自动化必须服从 L1 / L2 / L3 授权矩阵。即使未来创建自动化，也不能绕过 L3 单独审批。

## 12. 禁止事项

- 不能因为有权限就新增 skill / harness。
- 不能把项目经验直接写成长期规则。
- 不能把支线任务当主线做。
- 不能为了清理而删除未审证据。
- 不能把临时 staging / commit 材料当 stable 文档。
- 不能把用户的大方向讨论直接落成 stable policy。
- 不能把 L2 或 L3 动作伪装成 L1 自动动作。

## 13. 和现有治理结构的关系

本机制不是推翻已有治理，而是作为上位调度层，把已有治理零件串起来。

| 已有治理 | 在本机制中的位置 | 作用 |
|---|---|---|
| A1 PRD 主链路 | Stable Core | 固定 PRD 输出底线，避免 PRD 退化。 |
| A2 workflow / pipeline / harness 合同 | 治理部门 | 保证声明的 workflow 和实际 pipeline 不再脱节。 |
| A3 必要检查 | 质量部门 | 作为最低回归和真实输出检查。 |
| A4 closeout / 偏好边界 | 学习部门 / 归档部门 | 控制项目经验、偏好和长期记忆边界。 |
| B 批治理文档 | 制度文件 | 提供操作说明、审计记录和分区依据。 |
| C 批项目产物 | 项目生命周期管理 | 判断项目 active / closeout / archive / delete-after-30-days。 |
| D 批 candidate 能力 | Candidate Sandbox | 保留候选能力，但不自动转 stable。 |
| E 批 root / archive | 归档部门 | 控制历史资产、archive 证据和删除候选。 |
| `docs/thread_registry.md` | 上下文恢复 / 监察台账 | 防止任务漂移、线程停滞和上下文丢失。 |

## 14. 试运行验收标准

建议先试运行 1-2 周，不直接 stable。试运行期间用这些标准判断是否有效：

| 指标 | 目标 |
|---|---|
| 主线恢复 | 换线程或上下文稀释后，能在 1 分钟内从 `thread_registry.md` 找回当前主线。 |
| 任务漂移 | 新方向不会直接打断当前主线，先进入分类和看板。 |
| 工具最小化 | 不新增 skill / harness / automation，除非经过 proposal 和用户批准。 |
| 低风险执行 | 文档审查、分类、看板更新、check-only 能顺畅执行。 |
| 高风险上报 | stable、长期记忆、删除、归档、push / PR 都能被清楚列为拍板项。 |
| 授权分级 | L1 自动、L2 汇总审批、L3 单独审批能被清楚区分。 |
| 复盘反哺 | 项目 closeout 和用户反馈能进入 architecture inbox 或 proposal，而不是直接长期化。 |
| 瘦身意识 | 每次项目结束、版本变化或线程停滞，都能产生 archive / delete-after-30-days 候选。 |

如果试运行达不到这些标准，先修本 proposal 或台账字段，不新增工具。

## 15. 当前落地方式

先采用轻量落地：

1. `docs/thread_registry.md` 维护上下文恢复卡、治理任务看板和优化迭代流程。
2. 本 proposal 作为上位机制候选。
3. 按 L1 / L2 / L3 授权矩阵试运行：L1 自动处理，L2 批量审批，L3 单独审批。
4. 继续处理 C 批项目产物、archive、zip、raw 和 candidate 瘦身。
5. 观察 1-2 周后，用户再决定是否写入 stable operating model 或创建低风险自动化。

## 16. 需要用户后续拍板

| 决策 | 我的建议 | 效果 |
|---|---|---|
| 本 proposal 是否转 stable | 先试运行，不直接 stable | 降低过早固化风险。 |
| 是否创建 5 天停滞自动化 | 先不创建 | 先用台账观察一致率。 |
| 是否允许 L1 自动更新台账和 proposal 分类 | 试运行允许 | 减少用户低价值确认，让上下文持续可恢复。 |
| 是否允许 L2 汇总审批 | 允许 | 用户不再每一步把关，只审核批量包。 |
| 是否新增工具支持看板 | 不新增 | 先用 Markdown，成本最低。 |
| 是否把部门制写入 `agent.md` | 暂缓 | 当前已有任务漂移规则，先试运行上位 proposal。 |

## 17. 本轮不做

- 不修改 stable docs。
- 不新增 skill。
- 不新增 harness。
- 不新增 workflow stage。
- 不新增 automation。
- 不删除、归档、提交项目产物。
- 不写长期记忆。
