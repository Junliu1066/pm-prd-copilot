# 模型更新快速适配机制方案

- 状态：proposal / 已获方向确认，尚未转 stable policy
- 日期：2026-05-05
- 目标：让 agent 架构在模型更新时最快速适配，而不是在模型更新后做一次大而全的治理体检。
- 边界：自动化只负责触发、检测、报告；不自动修改文件、不新增 skill / harness、不修改 stable 规则、不删除、不提交。

## 1. 机制目标

模型更新后，系统应先判断更新等级，再用最小成本判断哪些 skill / harness / prompt / eval 需要调整，最后只把必要改动交给用户拍板。

这套机制要解决的问题：

- 新模型出现后，旧 prompt 和旧 harness 不能限制新模型能力。
- 模型能力变化后，系统能快速知道哪些输出链路需要复核。
- 不因为模型更新就触发全仓库治理审计。
- 不因为模型更新就自动重构架构。
- 用户看到的是“该改什么、为什么、最小怎么改”，而不是一堆噪音。

## 2. 模型更新轻量哨兵

职责：

- 每天轻量检测模型 / 平台更新。
- 只识别新增、下线、重命名、能力边界、上下文、工具调用、多模态、输出格式、成本和弃用风险变化。
- 不做治理审计。
- 不自动改文件。
- 不新增 skill / harness。
- 不修改 stable 规则。

输出：

- 未发现更新：只输出简短结论。
- 发现更新：进入模型更新分级，并生成后续报告。

## 3. 模型更新分级

| 等级 | 定义 | 动作 |
|---|---|---|
| L0 | 无关更新，不影响当前 agent 架构 | 只记录，不触发复核 |
| L1 | 兼容性更新，例如命名、价格、小能力说明变化 | 生成模型能力差异卡 |
| L2 | 能力更新，例如推理、上下文、工具调用、多模态、结构化输出变化 | 生成差异卡 + Skill / Harness 适配矩阵 + 快速回归 |
| L3 | 破坏性更新，例如模型下线、API 不兼容、输出格式变化导致 pipeline 可能失败 | 触发 P0/P1 检查，并要求人工拍板 |

判断原则：

- 能不升级就不升级。
- 不确定时按更保守等级处理，但不能自动修改文件。
- L2 / L3 才允许触发快速回归或适配矩阵。

## 4. 模型能力差异卡

每次发现 L1 及以上更新时，生成差异卡。

必须包含：

- 新旧模型差异。
- 能力变化。
- 工具调用变化。
- 上下文变化。
- 推理变化。
- 输出格式变化。
- 成本变化。
- 弃用风险。
- 不确定项。
- 需要手测项。

差异卡只做事实和影响判断，不写稳定规则。

## 5. Skill / Harness 适配矩阵

当更新达到 L2 / L3 时生成。

检查对象：

- skill 指令是否还适合新模型。
- 旧 prompt 是否限制新模型能力。
- harness 测试规则是否过时。
- regression check 是否需要更新。
- 旧约束、假设、边界是否需要放松或重写。
- 哪些能力需要重新手动测试。

分级：

| 等级 | 定义 | 示例 |
|---|---|---|
| P0 | 必须马上改，否则跑不动 | 模型名下线、API 字段不兼容、结构化输出失败 |
| P1 | 旧规则可能限制新模型能力 | prompt 过度保守、关键词式 harness 误杀更好输出 |
| P2 | 可优化，但不急 | token 上限、eval 样例、风格提示可更新 |
| P3 | 观察即可 | 无明显影响或缺证据 |

## 6. 快速回归样例

模型更新达到 L2 / L3 时，使用固定小样本验证影响。

覆盖范围：

- PRD 输出。
- Codex 开发文档。
- 原型图层。
- AI 方案。
- B 包脱敏。
- harness check-only。
- 长上下文任务。
- 工具调用任务。

原则：

- 样例要小，优先判断是否退化。
- 不扩大成完整回归矩阵。
- 发现问题再做定向复核。

## 7. 最小补丁原则

模型更新后的修复顺序固定为：

1. 配置。
2. prompt wording。
3. 测试阈值。
4. eval 样例。
5. 局部调整 skill / harness。
6. 最后才考虑结构变化。

禁止默认动作：

- 不因模型更新而重构。
- 不新增 skill / harness，除非旧机制完全覆盖不了。
- 不删除旧约束，除非用户确认。
- 不把候选能力转 stable。

## 8. 人工拍板边界

以下事项必须人工确认：

- 改 stable 规则。
- 删除旧约束。
- 新增 skill / harness。
- 更换模型供应商。
- 成本明显变化。
- 长期规则调整。
- 大规模重构。

系统可自动完成：

- 发现更新。
- 分级。
- 生成差异卡。
- 生成适配矩阵。
- 跑轻量快速回归。
- 输出最小补丁建议。

系统不能自动完成：

- 写入稳定规则。
- 修改长期 skill / harness 行为。
- 接受高风险兼容性问题。
- 扩大治理范围。

## 9. 统一输出物

模型更新触发后，报告统一包含：

1. 模型更新分级结论。
2. 模型能力差异卡。
3. Skill / Harness 适配矩阵。
4. 快速回归结果。
5. 最小补丁建议。
6. 需要人工拍板事项。

如果没有模型更新，不输出这些长报告，只输出轻量结论。

## 10. 与现有自动化的关系

- `模型更新轻量哨兵`：独立保留，负责触发和报告。
- `每日架构治理、AI 情报与教学总结`：只引用哨兵结果，不自行判断模型更新，不自行发起 skill / harness 复核。
- `凌晨错误巡检与 Bug 记录`：只检查哨兵活性与配置健康，不执行模型更新判断。

## 11. 当前建议

本方案可以作为模型更新适配的长期候选机制先运行 2-4 周。

暂不直接转 stable policy。观察项：

- 哨兵是否误报。
- 哨兵是否漏报。
- L0-L3 分级是否符合用户判断。
- 输出是否足够短。
- 适配矩阵是否真的减少返工。

转 stable 前必须再次由用户批准。
