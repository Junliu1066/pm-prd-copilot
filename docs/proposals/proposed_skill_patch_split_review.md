## 拆分判断

### 1. 市场默认值来源

原始反馈：如果未说明市场，竞品分析默认中国市场。

我的建议：不进入通用 Skill 的稳定规则，先保持为项目偏好或用户长期偏好候选。

推荐处理方式：

- 当前请求明确指定市场：按当前请求执行。
- 当前项目缓存有已批准市场偏好：按项目缓存执行。
- 长期偏好有已批准市场偏好：可以作为默认值，但输出中必须标明来源。
- 都没有：标记 `market_scope: unresolved`，不要自行假设。

优势：

- 避免把“中国市场”写死进通用 Skill。
- 避免跨项目误用市场假设。
- 仍能支持你在具体项目里的默认市场偏好。

劣势 / 风险：

- 每个新项目如果没设偏好，可能需要多问一次。
- 如果项目缓存不完善，市场判断会更保守。

推荐结论：`keep_project_or_user_preference_candidate`。暂不改 Skill。

### 2. 竞品外延覆盖直接和间接影响

原始反馈：竞品不能只看垂类直接业务，只要在行业赛道内影响用户决策、注意力、时间和工作流，都应纳入竞品或替代方案。

我的建议：方向正确，但不要做成“每个项目必须列满所有类别”。更稳的表达是：竞品分析需要建立“竞品影响地图”，按项目复杂度覆盖直接工具、相邻平台、平台原生替代、内容/社区替代、线下服务、手动作业等类别；无关类别可以显式标记不适用。

优势：

- 能避免竞品分析过窄。
- 有利于发现平台原生能力、内容平台、人工流程对产品机会的影响。
- 能反哺 MVP 范围判断，避免和大平台硬碰硬。

劣势 / 风险：

- 小项目会增加分析成本。
- 如果没有证据来源，容易把假设写得像事实。
- 如果强制所有类别，会让 PRD 变重。

推荐结论：`promote_later_after_output_shape_review`。建议下一轮针对 `competitor-gap-analyzer` 做最小 wording 修复，但先不动本轮。

### 3. PRD 分析视觉化

原始反馈：用户群体判断可以在第一段做矩阵图；方案和 MCP/Skill 分工需要流程图和图标呈现。

我的建议：不能按原文整体应用。A1 已确认的稳定口径是：PRD 必须有页面说明、页面跳转关系、原型图层；其他图表按项目必要性放在对应章节。Skill/MCP 分工图属于内部治理或复杂方案审查，不应默认进入普通 PRD。

更稳的拆分方式：

- 用户群体广泛时：可以输出用户矩阵或优先级矩阵。
- 竞品范围复杂时：可以输出竞品影响地图。
- 流程复杂时：可以输出流程图或泳道图。
- 涉及内部治理、AI 方案、Codex 开发链路时：才输出 Skill/MCP/职责分工图。
- 普通 PRD 不默认暴露内部 Skill/MCP 结构。

优势：

- 保留视觉化辅助理解的价值。
- 避免 PRD 再次退化为集中式“可视化层”。
- 避免向外部读者暴露内部治理机制。

劣势 / 风险：

- 需要模型判断“何时必要”，短期可能还会有漏图或多图。
- 需要通过 real-output eval 持续校准。

推荐结论：`keep_as_conditional_output_rule`。暂不改 Skill，后续通过 PRD 输出样例继续校准。
