# 线程生命周期监督方案

- 日期：2026-05-02
- 状态：proposal，不是 stable policy。
- 目标：把长期搁置的对话线程纳入治理生命周期，做到 5 天未推进自动总结、向用户汇报是否继续 / closeout / 归档 / 清理，同时把用户反馈中的经验教训提炼为候选，不自动写长期规则。
- 原则：
  - 主线任务优先，禁止任务漂移。
  - 线程经验可以自动总结，长期规则必须用户批准。
  - 不新增 skill，不新增 harness。
  - 不自动归档、删除、提交、push、PR、写长期记忆或修改稳定架构。

## 一句话方案

每个重要线程都必须有主线、有状态、有停滞判断、有收口出口。

当一个线程 5 天没有推进时，系统不自动清理，而是生成“线程停滞报告”，让用户决定：

```text
继续推进 / 主动暂缓 / closeout / 归档候选 / 30 天后删除候选
```

## 要解决的问题

| 问题 | 当前表现 | 线程治理后的效果 |
|---|---|---|
| 线程越开越多 | 多个 PRD、原型、开发、治理、归档线程同时存在，容易忘记停在哪里。 | 每个线程有状态和下一步最小动作。 |
| 搁置后经验丢失 | 5 天或更久未处理后，需要重新翻上下文。 | 自动生成停滞总结和待拍板项。 |
| 项目偏好误长期化 | 某个项目里的反馈可能被误当作通用规则。 | 经验分成线程经验、项目经验、长期候选。 |
| 支线任务影响主线 | 在治理线程里展开项目业务功能，导致跑偏。 | 每个线程先写 current main task 和 out-of-scope。 |
| 清理瘦身做不下来 | 不知道哪些线程能归档，哪些还要继续。 | 线程 closeout 和项目 closeout 对齐，逐步形成清理候选。 |

## 线程状态

| 状态 | 含义 | 推荐动作 |
|---|---|---|
| `active` | 正在推进，主线清楚。 | 正常处理当前主线。 |
| `waiting_user` | 卡在用户拍板。 | 汇报选项、优劣势、建议和不处理的风险。 |
| `parked` | 用户或系统主动暂停。 | 保留现场，不频繁提醒；到期再汇报。 |
| `stale_5d` | 5 天没有推进。 | 自动生成线程停滞报告，问是否继续、closeout 或归档候选。 |
| `closeout_candidate` | 主任务基本完成或长期暂停。 | 总结经验、问题、文件产物、可反哺架构内容。 |
| `archived` | 已归档，不再作为 active 上下文读取。 | 只保留归档记录和已确认沉淀。 |
| `delete_after_30_days_candidate` | 归档后可清理候选。 | 30 天后拿精确清单再次让用户批准。 |

状态不确定时，默认 `active` 或 `waiting_user`，不能默认归档或删除。

## 5 天停滞报告

当线程 5 天未推进时，生成报告草案，必须包含：

```text
线程名称：
线程类型：治理 / PRD / 原型 / UI / 开发 / 归档 / AI 情报 / 其它
所属项目：
最后活跃时间：
当前主线：
预期效果：
当时为什么停住：
已完成事项：
未完成事项：
需要用户拍板：
涉及文件 / 产物：
发现的问题：
用户在本线程中教过的内容：
经验教训候选：
可反哺架构候选：
清理或归档候选：
我的建议：继续 / 暂缓 / closeout / 归档候选 / 30 天后删除候选
不同选择的效果：
下一步最小动作：
```

报告必须中文输出。

## 经验教训分层

线程停滞总结不能直接写长期记忆。经验必须先分层：

| 层级 | 含义 | 处理方式 |
|---|---|---|
| 线程经验 | 只解释这个线程发生了什么。 | 可以自动总结到停滞报告。 |
| 项目经验 | 只对当前项目有效，例如 UI 风格、市场默认、业务偏好。 | 项目内保留，closeout 前对齐。 |
| 长期候选 | 可能适用于所有项目，例如禁止任务漂移。 | 进入 proposal / architecture inbox，必须用户逐条批准。 |
| 不应吸收 | 一次性情绪、临时试验、过期业务判断。 | 明确标记不吸收。 |

示例：

| 用户反馈 | 分类 | 原因 |
|---|---|---|
| “现在主线是治理修复，食物热量拍摄不要展开。” | 长期候选 / 已批准方向 | 这是禁止任务漂移的通用规则。 |
| “fitness 默认中国市场。” | 项目经验 | 只适用于 fitness 项目，不应跨项目。 |
| “暗黑机械风。” | 项目经验 | 是 fitness 原型风格，不是全局 UI 默认。 |
| “报告都给中文。” | 长期偏好候选 / 已批准方向 | 用户明确长期要求，适合进入长期偏好。 |

## 线程台账建议

要监督线程，需要轻量台账。建议先 proposal，不马上落地。

候选路径：

```text
docs/thread_registry.md
```

建议字段：

| 字段 | 作用 |
|---|---|
| `thread_name` | 线程名称。 |
| `thread_type` | 治理 / PRD / 原型 / UI / 开发 / 归档 / AI 情报。 |
| `project_id` | 所属项目，没有则写 governance。 |
| `main_task` | 当前主线。 |
| `expected_effect` | 完成后要达到的效果。 |
| `status` | active / waiting_user / parked / stale_5d / closeout_candidate / archived。 |
| `last_active_at` | 最后推进日期。 |
| `blocked_by` | 当前阻塞点。 |
| `pending_user_decisions` | 需要用户拍板事项。 |
| `artifacts_touched` | 涉及文件或目录。 |
| `lessons_candidate` | 可沉淀经验候选。 |
| `cleanup_candidate` | 可清理候选。 |
| `next_minimal_action` | 下一步最小动作。 |

为什么先用 Markdown：

- 轻量。
- 容易人工审核。
- 不需要新增脚本。
- 后续如果稳定，再考虑 YAML 或自动化。

## 和现有治理体系的关系

| 现有文件 / 机制 | 如何嵌入线程治理 |
|---|---|
| `agent.md` | 已有主线任务和禁止任务漂移规则，线程报告必须引用 current main task。 |
| `docs/operating_model.md` | 后续可加入线程停滞检查节奏。 |
| `docs/project_lifecycle.md` | 项目线程、归档线程、长期治理线程继续分离。 |
| `docs/daily_report_template.md` | 后续可增加“停滞线程与待拍板项”章节。 |
| `docs/architecture-inbox/` | 线程经验先作为架构候选，不直接长期化。 |
| `memory-cache/` | 项目线程偏好只在项目内生效。 |
| `docs/error_reports/` | 如果线程停滞原因是 bug、遗漏或检查失败，记录为 bug / watch。 |
| 定时任务 | 后续可每天或每周检查 5 天未推进线程，但只汇报不处理。 |

## 自动化阶段

### 阶段 1：人工台账

当前推荐。

动作：

- 先写本方案。
- 后续经用户批准后建立 `docs/thread_registry.md`。
- 每次线程收口时人工更新台账。

不做：

- 不创建自动化。
- 不扫描所有历史线程。
- 不自动归档。

### 阶段 2：提醒和总结草案

前提：

- 台账已运行一段时间。
- 用户确认字段和分类有用。

动作：

- 定时检查 `last_active_at`。
- 超过 5 天生成停滞报告。
- 把报告放入每日 / 每周汇报。

仍不允许：

- 自动归档。
- 自动删除。
- 自动写长期记忆。
- 自动提交。

### 阶段 3：低风险半自动

前提：

- 连续 2-4 周判断一致率高。

允许：

- 自动刷新线程状态。
- 自动生成 closeout 草案。
- 自动列待拍板项。
- 自动汇总经验候选。

永久禁止自动：

- 归档线程。
- 删除文件。
- 清空项目缓存。
- 写长期记忆。
- 修改 skill / harness / workflow / registry。
- git add / commit / push / PR。

## 需要用户拍板

| 拍板项 | 我的建议 | 不同选择的效果 |
|---|---|---|
| 是否采用线程生命周期监督机制 | 先作为 proposal 使用 | 能先统一判断，不会立刻增加稳定规则。 |
| 是否现在创建 `docs/thread_registry.md` | 下一步再创建 | 先审方案更稳；现在创建也可行，但会新增治理文件。 |
| 5 天是否作为默认停滞阈值 | 采用 5 天 | 足够及时，又不会太频繁打扰。 |
| 是否创建自动化提醒 | 暂不创建 | 先人工台账校准；直接自动化可能误报。 |
| 停滞报告是否进入每日汇报 | 后续加入 | 能减少漏项，但应先确认报告格式。 |
| 经验候选是否自动写长期记忆 | 不允许 | 长期记忆必须逐条批准。 |

## 推荐落地顺序

1. 当前先保留本方案为 proposal。
2. 如果用户认可，下一步创建轻量 `docs/thread_registry.md` 草案。
3. 用当前治理线程和 2-3 个项目线程试跑 1-2 周。
4. 再决定是否更新 `docs/operating_model.md` 和 `docs/daily_report_template.md`。
5. 最后再决定是否创建 5 天停滞自动提醒。
