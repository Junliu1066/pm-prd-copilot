# Learning Feedback Rules

## Purpose

Capture reusable user corrections without silently changing stable memory, skills, or workflow behavior.

## Capture When

Capture a learning signal when the user:

- Corrects a repeated behavior.
- States a stable preference.
- Defines a reusable workflow rule.
- Rejects a document structure for future use.
- Explains a project governance expectation.

## Do Not Capture As Stable Memory When

- The instruction appears one-off or project-specific.
- The user is brainstorming rather than deciding.
- The preference conflicts with existing stable rules and has not been approved.
- The lesson would expose internal governance details in user-facing artifacts.

## Proposal Format

When a reusable lesson should be proposed, record:

- User feedback
- Proposed interpretation
- Lesson type
- Affected components
- Expected future behavior
- Verification hint
- Whether explicit approval is required

## Current Reusable Signals

For development artifact work, treat these as candidate reusable behaviors unless superseded:

- If the user says a demo is already running, do not frame core feasibility as uncertain.
- Keep internal stewardship, review, efficiency, and learning controls out of ordinary user-facing project artifacts unless asked.
- Use a final development document as the implementation source of truth when multiple staged documents exist.
- Add review gates for product scope, technical contracts, security, tests, efficiency, and learning when a project spans multiple phases.
