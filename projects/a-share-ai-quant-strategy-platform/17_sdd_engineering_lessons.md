# SDD 工程化经验沉淀

- 文档状态：Draft / 项目复盘沉淀
- 适用项目：A股 AI 量化策略研究平台
- 适用读者：产品负责人、研发负责人、Codex 操作者、代码集成人、QA
- 最后更新时间：2026-05-10

---

## 1. 这轮最重要的结论

文档到工程可用之间，差的不是内容数量，而是能不能被执行、检查、约束和回滚。

本项目经历了三个阶段：

| 阶段 | 状态 | 典型产物 |
|---|---|---|
| 蓝图级 | 知道要做什么 | PRD、原型、内部开发文档 |
| 工程启动级 | 能开始建工程 | API 规格、数据库设计、后端规范、任务拆分、测试计划 |
| 多线程可控级 | 能让多个 Codex 线程并行 | `AGENTS.md`、Task Brief、文件边界矩阵、失败协议、可执行门禁 |

真正适合 Codex 多线程开发的是第三层。

---

## 2. PRD / SDD / 开发文档不是终点

PRD 解决“为什么做、给谁做、做什么”。  
SDD 解决“系统怎么设计、模块怎么拆、数据怎么流”。  
工程可用包还必须解决：

- 线程进入项目后先读什么。
- 哪些文件能改，哪些不能改。
- 质量门禁怎么执行。
- 失败后怎么停、怎么修、谁决策。
- 契约漂移怎么处理。
- 合并前怎么判断可以过。

所以工程包不能只停在文档说明层，必须补执行层。

---

## 3. 质量门禁必须是命令

“需要检查合规、接口、原型”这类描述不够硬。

门禁必须变成命令：

```bash
make check-all
```

并且必须满足：

- 成功返回 0。
- 失败返回非 0。
- 输出能定位问题。
- Codex 交付摘要必须写执行结果。

本项目当前门禁：

```bash
make check-docs
make check-compliance
make check-contracts
make check-prototype
make check-all
```

后端 / 前端工程创建后，再把 `./mvnw test`、`npm run lint`、`npm run build` 接入统一门禁。

---

## 4. 门禁脚本本身也需要调试

第一版合规脚本暴露了两个问题：

- 单纯扫关键词会误伤，例如普通英文单词里包含旧收费层级片段。
- 风险词在“禁止 / 不做范围 / 拦截”语境中是合理的，不能一律判失败。

经验：

- 旧收费词要按独立词匹配，避免误伤普通英文单词。
- 高风险词要区分正向能力描述和禁止语境。
- 脚本要允许“禁止、不得、不展示、不返回、拒绝、拦截、检查”等语境。
- 脚本不能扫到自己内部的检测正则后误报。

质量门禁不是一次写完的，它也是工程资产，需要迭代。

---

## 5. 分线程的核心不是并行，而是边界

多开 Codex 线程不等于效率更高。没有边界时，并行会制造冲突。

必须先回答：

- 谁能改 API？
- 谁能改数据库？
- 谁能改原型？
- 谁能改公共后端代码？
- 谁能改质量门禁？
- 哪些文件只读？
- 哪些路径必须锁？
- 线程越界怎么办？

本项目用三层治理解决：

- `12_codex_thread_governance.md`：线程生命周期和合并治理。
- `14_file_boundary_matrix.md`：文件权限矩阵。
- `15_failure_handling_protocol.md`：越界和失败处理。

---

## 6. Task Brief 必须硬

软任务示例：

```text
做积分模块。
```

这种任务会导致 Codex 自己决定范围，容易越界。

硬 Task Brief 必须包含：

- branch
- owner
- goal
- allowed paths
- forbidden paths
- locked paths
- input docs
- contract touched
- exact commands
- exit criteria
- failure action

如果任务没有 Task Brief，不应开工。

---

## 7. 原始需求记录不能当当前事实

`00_source_notes.md` 保存了历史输入，其中可能包含已经被替代的想法。

经验：

- source notes 是历史材料，不是当前事实。
- 当前事实以最新用户明确指令、PRD、内部开发文档和工程契约为准。
- Codex 新线程不能从 source notes 里把旧收费制度、旧商业模式或旧范围重新带回来。

对于本项目，当前事实是：

- C 端统一积分制度。
- 不使用旧的会员、月票、观众票和订阅制表达作为主收费制度。
- 量化默认 `research_only` / `simulation_only`。
- 不接实盘、不接券商账户、不做跟单、不展示买卖点和实时信号。

## 8. 失败协议比事后解释更重要

工程开发一定会失败：

- 测试失败。
- 合规检查失败。
- API 漂移。
- migration 冲突。
- 文件越界。
- 线程失控。
- 上下文污染。

没有协议时，Codex 往往会继续扩大修改范围。

正确方式：

```text
先停止
-> 记录失败
-> 判断归属
-> 最小修复
-> 复跑同一门禁
-> 再交付
```

禁止为了保留已有工作量而降低门禁。

---

## 9. 契约文件要先于实现稳定

前后端并行时，最容易乱在 API 和数据库。

经验：

- `06_api_spec.md` 是 API 真相来源。
- `07_database_schema.md` 是数据库真相来源。
- `08_backend_engineering_spec.md` 是后端实现真相来源。
- 实现线程发现契约不够用时，应提交变更请求，不应先写代码。
- 前端线程只能消费接口契约，不应临时发明字段。

这能减少联调返工。

## 10. 工程可用要分层判断

以后判断文档是否工程可用，可以用这张表：

| 层级 | 判断标准 |
|---|---|
| 蓝图级 | 能解释产品和模块，但不能直接开工 |
| 工程启动级 | 有 API、库表、工程规范、任务拆分和测试计划 |
| 多线程可控级 | 有 Agent 指令、可执行门禁、Task Brief、文件边界、失败协议和执行手册 |
| 上线可发布级 | 有真实实现、CI、监控、发布、回滚、合规确认和生产配置 |

当前项目已经补到“多线程可控级”的文档和轻量门禁层。  
后续进入“上线可发布级”还需要真实代码、CI、监控、生产配置和法务 / 合规确认。

## 11. 下一次类似项目的推荐顺序

建议顺序：

```text
PRD
-> 原型
-> 内部开发文档
-> API 规格
-> 数据库设计
-> 后端工程规范
-> 任务拆分
-> 测试计划
-> 本地运行手册
-> 分线程治理
-> AGENTS.md
-> 可执行门禁
-> Task Brief 模板
-> 文件边界矩阵
-> 失败处理协议
-> Codex 执行手册
```

不要等开始并行开发后再补边界和失败协议。
