# AI 协作效率平台 - 引入业务思维解释文档

- 文档状态: Draft
- 生成日期: 2026-05-08
- 文档用途: 单独解释为什么本平台必须引入业务思维，以及业务思维应该进入哪些产品与工程环节
- 适用读者: 业务负责人、产品经理、研发负责人、评审人、HR、AI 平台负责人
- 说明: 本文是独立解释文档，不替代 PRD、原型图和开发文档。

## 1. 核心结论

AI 协作效率平台的核心不是“记录员工用了多少次 AI”，而是判断 AI 是否真实提升了业务效率、产出质量、组织复用和风险治理能力。

因此，平台必须引入业务思维。

如果没有业务思维，平台会退化成调用次数统计、耗时估算工具或简单积分系统。这样的系统很容易出现三个问题:

- 高价值任务和低价值任务被同等对待。
- 会用 AI 的人不一定真正创造业务价值，却可能通过高频低价值任务刷分。
- 评分看似精确，但无法解释为什么这个任务值得更多分、为什么这个 Skill 应该更高权重、为什么某些产出必须进入 Review。

正确边界应该是:

```text
业务定义什么叫价值
产品把价值规则产品化
工程把规则计算准确、保存准确、解释准确
数据用真实样本持续校准规则
```

加密、授权、充值同样需要业务思维。它们不能按“先卖额度 / 先充值”设计，而应按业务可解释顺序设计:

```text
审计 -> 授权 -> 版本控制 -> 风险分级 -> 价值计量 -> 成本归集 -> 结算 / 充值 / license
```

原因是: 没有审计就不知道发生了什么；没有授权就不知道谁能做什么；没有版本控制就无法知道消耗了哪个能力资产；没有风险分级就无法判断能否放行；没有价值计量就无法证明资产有贡献；没有成本归集就无法知道资源用了多少钱；最后才谈内部收费、外部收费、充值或 license。

这里的加密不是只加密附件或密钥，而是保护 Skill 源码 / 能力实现包。别人可以被授权调用某个 Skill，但不能因此拿到 Prompt 源、Workflow DAG、工具适配代码、执行脚本、私有规则或测试集答案。源码查看和源码导出必须是独立授权。

## 2. 为什么业务思维是核心

### 2.1 AI 效率不是单纯节省时间

同样节省 2 小时，业务价值可能完全不同:

| 场景 | 节省时间 | 业务价值判断 |
|---|---:|---|
| 整理一份普通会议纪要 | 2 小时 | 有效率价值，但风险和业务影响较低 |
| 发现客户流失原因 | 2 小时 | 可能影响续约和收入，应更高权重 |
| 初筛合同风险 | 2 小时 | 涉及合规和损失控制，需要 Review 和更高风险权重 |
| 生成对外发布材料 | 2 小时 | 可能影响品牌和客户承诺，必须强审 |

所以，贡献分不能只看时间差。业务价值、风险等级、产出质量、复用价值都必须进入计算。

### 2.2 Skill 调用成功不等于业务有效

一个 Skill 调用成功率很高，但如果只是在做格式整理、简单摘要或重复搬运，它对组织的核心价值有限。

反过来，一个高风险、高复杂 Skill 调用频次不高，但能稳定完成经营分析、风险识别、客户复盘，它的业务价值可能更高。

因此 Skill 权重不能只看调用次数或成功率，必须结合:

- 是否解决核心业务问题。
- 是否节省真实人工时间。
- Review 通过率是否稳定。
- 输出是否能直接进入业务流程。
- 是否能沉淀为组织知识。
- 风险命中率是否可控。

### 2.3 Review 不是形式审核，而是业务质量判断

AI 输出是否“好”，不能只由模型置信度判断。业务侧需要判断:

- 结论是否覆盖业务问题。
- 数据口径是否符合部门标准。
- 建议是否能执行。
- 是否遗漏关键风险。
- 是否可以复用给其他同类任务。
- 是否符合客户、合规、财务、对外发布要求。

Review 的本质是把业务质量标准结构化。

## 3. 业务思维应该进入哪些环节

### 3.1 任务分类

任务分类决定平台如何理解一项工作。

需要业务参与定义:

- 哪些任务类型属于高频效率任务。
- 哪些任务类型属于高价值分析任务。
- 哪些任务类型属于高风险任务。
- 哪些任务类型可以自动通过。
- 哪些任务类型必须进入 Review。

示例:

| 任务类型 | 业务判断 | 平台处理 |
|---|---|---|
| 会议纪要结构化 | 高频、低风险、可复用 | 可低风险自动通过 |
| 月度经营分析报告 | 中高价值、需口径一致 | P1 或高价值 Review |
| 合同风险初筛 | 高风险、需专业判断 | 强制 Review 或 P0 拦截 |
| 对外客户方案 | 高影响、高风险 | 强制 Review，必要时多人评审 |

### 3.2 基准工时

基准工时不是员工随便填，也不是工程拍脑袋。

业务需要参与:

- 给出任务正常人工耗时范围。
- 提供历史样本。
- 判断哪些任务因材料量、复杂度、跨部门协作而耗时更长。
- 定期修正不合理基准。

推荐口径:

```text
基准工时 = 任务类型默认值 + 复杂度修正 + 历史样本校准 + 主管抽样复核
```

工程落地要求:

- 保存基准来源。
- 保存样本窗口。
- 保存配置版本。
- 员工自报必须与系统记录、历史样本或主管复核区分开。

### 3.3 业务价值权重

业务价值权重回答的问题是: 同样完成一项任务，为什么有的任务贡献更高。

建议分层:

| 业务价值类型 | 示例 | 建议权重 |
|---|---|---:|
| 内部格式整理 | 排版、轻量摘要 | 0.70 |
| 常规内部运营 | 周报、普通会议纪要 | 1.00 |
| 客户、收入、成本相关 | 客户复盘、报价分析、预算分析 | 1.10 |
| 合规、合同、财务风险 | 合同审查、财务异常核验 | 1.20 |
| 公司级战略事项 | 经营策略、关键项目复盘 | 1.30 |

注意: 权重不是为了鼓励所有人都去抢高价值任务，而是让平台承认不同任务的真实业务影响。

### 3.4 任务复杂度

复杂度不是“文字多”或“文件多”这么简单。

业务需要定义哪些因素真正代表难度:

- 材料量。
- 口径歧义。
- 领域专业深度。
- 输出复杂度。
- 跨角色依赖。
- 风险等级。

建议映射:

| 复杂度维度 | 业务含义 |
|---|---|
| 材料量 | 数据表、附件、历史资料是否多 |
| 口径歧义 | 是否需要解释指标、财务口径、业务规则 |
| 领域深度 | 是否需要行业、客户、法务、财务知识 |
| 输出复杂度 | 是否要求图表、结论、行动项、复盘建议 |
| 跨角色依赖 | 是否需要销售、财务、法务、运营协同 |
| 风险等级 | 产出是否影响客户、合规、对外承诺 |

### 3.5 Review 标准

Review 评分维度必须来自业务对“好产出”的定义。

默认模板:

| 维度 | 权重 | 业务含义 |
|---|---:|---|
| 完整性 | 40% | 是否覆盖任务目标、证据、结论和行动项 |
| 准确性 | 25% | 数据、口径、推理和引用是否正确 |
| 可用性 | 20% | 是否能直接进入业务流程或少量修改后使用 |
| 复用价值 | 10% | 是否可沉淀为 SOP、Prompt、Case 或模板 |
| 合规安全 | 5% | 是否满足权限、隐私、对外发布和风控要求 |

高风险场景中，合规安全可以不是 5%，而是硬门槛。例如合规安全低于 60 分时，即使总分达标，也应该复审或拦截。

### 3.6 Skill 权重

Skill 权重必须结合业务判断和系统数据。

员工不参与 Skill 权重、风险等级、Review 规则、Prompt 模板、上架、下线或回滚管理。员工的角色是使用 Skill，并把使用中的痛点反馈给管理角色。

系统可以计算建议值:

```text
Skill 建议权重 =
基础值
+ Review 通过率贡献
+ 节省工时贡献
+ 验证复用贡献
- 风险命中惩罚
- 执行失败惩罚
```

但最终生效权重必须经过业务或平台负责人审批。

业务审批时重点看:

- 这个 Skill 是否解决核心业务场景。
- 是否真的减少人工工作，而不是制造返工。
- 是否稳定通过 Review。
- 是否降低风险，还是放大风险。
- 是否产生可复用知识。
- 是否存在刷分空间。

员工反馈时重点看:

- 输出哪里不准。
- 哪些字段缺失。
- 哪些格式不符合业务要求。
- 哪些结果不可直接使用。
- 哪些步骤或参数太复杂。
- 是否没有明显节省时间。
- 是否希望支持新的任务场景。

### 3.7 封顶与反刷分

封顶规则必须来自真实业务工作方式。

需要业务判断:

- 哪些任务属于低价值高频。
- 哪些任务容易被拆分刷分。
- 哪些 Skill 输出价值有限但调用成本低。
- 哪些任务虽然高频，但对组织确实重要，不应该简单封顶。

示例:

| 场景 | 业务判断 | 处理 |
|---|---|---|
| 同一天重复生成类似摘要 | 低价值重复 | 日封顶或降权 |
| 把一个完整分析拆成 5 个小任务 | 拆分刷分 | 合并计分或异常复核 |
| 合同风险初筛多次调用 | 高风险但必要 | 不按低价值封顶，强制 Review |
| 会议纪要结构化高频调用 | 有效率价值但边际价值递减 | 设日封顶 |

### 3.8 知识沉淀

知识沉淀不是把所有 AI 输出都放进知识库。

业务需要判断什么值得沉淀:

- 可复用 Prompt。
- 可复用 SOP。
- 业务 Case。
- 失败模式。
- Review 经验。
- 标准分析模板。

判断标准:

- 是否被 Review 通过。
- 是否可被其他人复用。
- 是否有明确适用场景。
- 是否不会传播错误口径。
- 是否有版本和来源。

### 3.9 员工反馈与 Skill 优化

员工不应该直接参与 Skill 管理，但必须有反馈入口。

原因:

- 员工最了解真实使用场景和痛点。
- 员工直接管理 Skill 会带来配置混乱、刷分空间和风险放大。
- Skill 的生产配置需要版本、审批、测试、审计和回滚。

正确机制:

```text
员工反馈痛点
-> 系统结构化记录
-> 管理端聚合分析
-> Skill 管理员创建优化任务
-> 测试新版 Skill
-> 审批上架
-> 员工查看反馈处理状态
```

员工可做:

- 提交 Skill 使用反馈。
- 标记输出问题。
- 说明期望结果。
- 上传可选证据。
- 申请支持新任务类型。
- 查看自己反馈的处理状态。

员工不可做:

- 修改生产 Skill。
- 修改 Skill 权重。
- 修改 Prompt 模板。
- 修改风险等级。
- 修改 Review 规则。
- 上架、下线或回滚 Skill。

管理端需要把反馈和以下数据一起看:

- Review 退回率。
- 执行失败率。
- 平均节省工时变化。
- 风险命中率。
- 复用率。
- 同类反馈出现频次。

### 3.10 加密、授权、充值与成本归集

这部分尤其需要业务思维，因为它决定“能力资产如何被消费、谁有权消费、消耗如何归因、价值和成本如何解释”。

业务需要定义:

| 问题 | 业务需要给出的答案 | 平台落地 |
|---|---|---|
| 谁可以调用 | 哪些部门、角色、用户可以使用哪些 Skill / Workflow / Tool Permission | AuthorizationGrant |
| 什么资产可被授权 | Skill、Workflow、Knowledge Pack、Tool Permission、PromptVersion 的管理边界 | 资产层版本管理 |
| 哪些必须强审 | 哪些业务场景、风险等级、输出类型必须 Review | 风险规则和 Review 门禁 |
| 怎么算价值 | 哪些任务真正产生节省工时、质量提升、复用或业务影响 | ScoreRecord 和公式快照 |
| 怎么算消耗 | 模型 token、工具调用、存储、人工 Review、外部 API、license 成本是否计入 | UsageMeterRecord |
| 成本归给谁 | 归到部门、成本中心、项目、客户还是 license 池 | CostAllocationRecord |
| 怎么结算 | 内部成本分摊、外部客户收费、预充值、license 授权的规则 | SettlementAccount、RechargeTransaction、LicenseEntitlement |
| 怎么保护 Skill 源码 | 谁能调用、谁能查看源码、谁能导出源码 | SkillSourcePackage、source_read、source_export |
| 怎么保护敏感资产 | API Key、license key、客户材料、结算凭证由谁可见 | EncryptedSecret、脱敏和审计 |

产品约束:

- 员工是能力消费者，不是充值、license 或成本调整的管理者。
- 员工端只展示“可用 / 不可用”“额度不足 / 未授权 / license 过期”等可行动原因。
- 员工和外部客户只获得运行结果，不获得 Skill 源码、Prompt 源、Workflow DAG 或工具适配代码。
- 管理端先看计量证据和成本归集，再做充值、license 发放或冻结。
- 没有审计和计量证据的调用不能进入结算。
- 充值不是价值证明，license 也不是价值证明；价值仍然来自 Skill 调用后的业务结果、Review 和采用状态。

## 4. 业务、产品、工程的边界

| 角色 | 负责什么 | 不负责什么 |
|---|---|---|
| 业务负责人 | 定义价值、风险、复杂度、可用性、复用标准 | 不直接改数据库分数 |
| 产品经理 | 把业务规则产品化、配置化、可解释化 | 不用主观判断替代规则 |
| 工程团队 | 实现公式、版本、快照、审计、权限和计算服务 | 不自行定义业务价值 |
| 数据分析 | 做样本校准、异常检测、口径复盘 | 不替代业务审批 |
| HR / 主管 | 看趋势、做复核、提出校准建议 | 不把平台分数直接当绩效结论 |
| 员工 | 使用 Skill 并反馈痛点 | 不管理生产 Skill |

核心边界:

```text
业务参与规则定义和周期校准
不参与逐条随意修分
```

如果业务可以随意改每一条分数，平台会失去公信力。

如果业务完全不参与规则，平台会变成没有业务含义的技术积分系统。

## 5. 解决方案

### 5.1 总体方案

建立一套“业务规则配置中心 + 评分计算引擎 + Review 质量评审 + 数据校准闭环”的机制。

解决方案不是让业务人员逐条改分，而是让业务判断前置到规则层:

```text
业务规则配置中心
-> 任务分类、基准工时、业务价值、复杂度、Review 模板、封顶规则
-> 评分计算引擎按版本读取规则
-> 每次得分保存公式快照
-> 看板解释权重来源
-> 数据复盘发现规则偏差
-> 业务和产品按流程校准规则
```

最终效果:

- 业务价值可以进入分数。
- 分数仍然可计算、可复算、可解释。
- 业务不能随意改分。
- 工程不用替业务定义价值。
- 规则可以随业务变化持续校准。

### 5.2 组织机制方案

建立轻量的“AI 效率规则小组”，不新增复杂组织，但必须明确责任。

| 角色 | 参与动作 | 产出物 |
|---|---|---|
| 业务负责人 | 定义任务类型、业务价值等级、Review 标准 | 任务分类表、业务权重建议、Review 标准 |
| 部门主管 | 提供基准工时、复杂度判断、样本复核 | 基准工时样本、异常分数反馈 |
| 产品经理 | 把业务语言转成平台规则 | 配置项、流程、页面、验收标准 |
| AI 平台负责人 | 评估 Skill 价值、风险和上下线策略 | Skill 权重审批、Skill 生命周期策略 |
| 数据分析 | 提供历史样本、异常检测和影响预览 | 样本报告、规则调整影响分析 |
| 工程团队 | 实现规则配置、计算、快照、审计 | API、数据表、计算服务、看板解释 |
| HR | 约束分数使用边界，防止绩效误用 | 可见范围、申诉和复核规则 |

建议频率:

- 试点前集中定义首批规则。
- 试点前两周每周复盘一次。
- 稳定后每月复盘一次。
- 高风险规则调整前必须审批。

### 5.3 产品功能方案

平台需要增加或明确以下产品能力。

#### 5.3.1 业务规则配置中心

配置对象:

- 任务类型。
- 基准工时。
- 业务价值等级。
- 复杂度维度。
- Review 模板。
- Skill 权重。
- 低价值封顶规则。
- 风险分级规则。

每个配置必须包含:

- 适用部门。
- 适用任务类型。
- 生效时间。
- 版本号。
- 审批人。
- 调整原因。
- 是否影响历史分数。

原则:

- 规则变更只影响未来计算。
- 历史分数保留旧公式快照。
- 高风险规则必须二次确认。

#### 5.3.2 分数解释抽屉

看板每个分数旁边都应有“查看计算说明”。

解释内容:

- 任务类型。
- 基准工时来源。
- 实际耗时来源。
- 任务复杂度系数。
- 业务价值权重。
- Skill 权重。
- Review 质量系数。
- 复用系数。
- 反刷分系数。
- 封顶结果。
- 公式版本。
- 评审记录。

示例解释:

```text
本次贡献分 86.4 =
节省工时 7.13h
* 10 分 / 小时
* 任务复杂度 1.00
* 业务价值权重 1.00
* Skill 权重 1.15
* Review / 历史质量系数 1.054

未触发封顶，未命中重复任务。
```

#### 5.3.3 规则影响预览

业务或管理员调整规则前，系统先展示影响预览。

预览内容:

- 影响多少历史任务样本。
- 平均分变化。
- 高分任务变化。
- 低分任务变化。
- 哪些部门受影响。
- 哪些 Skill 权重变化最大。
- 是否可能导致刷分空间。

规则调整不应该直接上线，必须先看影响。

#### 5.3.4 申诉与复核

员工可以对分数发起申诉，但申诉处理应分类型:

| 申诉类型 | 处理方式 |
|---|---|
| 基准工时不合理 | 进入业务样本复核 |
| Review 评分争议 | 进入复审 |
| 任务复杂度错误 | 主管或评审人复核 |
| Skill 权重争议 | 不改单条分数，进入周期校准 |
| 系统计算错误 | 工程修复并保留调整记录 |

重点: 大多数争议应修正规则或复核事实，不应变成随意人工改分。

#### 5.3.5 Skill 使用反馈机制

员工侧提供轻量反馈入口，管理侧提供聚合和优化队列。

员工反馈类型:

- 输出不准确。
- 数据口径不一致。
- 缺字段或缺步骤。
- 格式不符合业务要求。
- 结果不可直接使用。
- 耗时没有明显降低。
- 风险提示不充分。
- 参数太复杂。
- 场景不匹配。
- 希望支持新任务类型。

反馈记录必须包含:

- Skill ID。
- Skill 版本。
- 关联任务 ID。
- 反馈类型。
- 严重程度。
- 说明文本。
- 可选附件。

管理端聚合维度:

- Skill。
- Skill 版本。
- 任务类型。
- 部门。
- 反馈类型。
- 严重程度。
- Review 退回率。
- 执行失败率。
- 风险命中率。

处理状态:

```text
已提交 -> 已分派 -> 分析中 -> 已纳入优化 / 不采纳 -> 新版已发布
```

边界:

- 反馈不直接改变 Skill。
- 反馈不直接加贡献分。
- 反馈可以成为 Skill 优化和业务规则校准的输入。
- 员工只能看自己的反馈明细，管理端看授权范围内聚合和必要明细。

#### 5.3.6 加密授权充值控制面

控制面页面按四层展示:

| 层级 | 页面展示 | 解决的问题 |
|---|---|---|
| 消费端 | 员工在 Cursor / Codex / Web 调用 Skill | 谁在用、用什么场景 |
| 控制面 | 审计、授权、风险、计量、成本、结算 | 谁能用、怎么放行、怎么归因 |
| 资产层 | Skill、Workflow、Knowledge Pack、Tool Permission、PromptVersion | 被消费的资产是什么版本 |
| 证据层 | SkillCall、ReviewRecord、ScoreRecord、UsageMeterRecord、AuditLog | 为什么可以计量、为什么可以结算 |

页面主链路:

```text
查看调用证据
-> 查看授权决策
-> 查看资产版本
-> 查看风险与 Review
-> 查看价值计量
-> 查看成本归集
-> 处理结算、充值或 license
```

产品要求:

- 充值和 license 不放在员工端。
- 充值页面必须展示来源计量和成本归集，不能只展示“加钱”。
- license 发放必须绑定资产范围、主体、有效期和配额。
- 成本调整和账期冻结必须写原因、审批人和审计。
- 敏感凭据只展示脱敏值，不展示明文。
- Skill 源码包只在运行时解密执行，不展示 Prompt 源、Workflow DAG、工具适配代码或执行脚本。

#### 5.3.7 Skill 并发执行机制

并发执行解决的是“复杂 Skill 处理慢”的问题，但不能牺牲业务一致性、风险控制和源码保护。

适合并发的业务场景:

| 场景 | 并发方式 | 汇总方式 |
|---|---|---|
| 多文档分析 | 每个文档一个子任务 | 汇总发现、去重、冲突检查 |
| 批量测试集 | 每个 Case 一个子任务 | 统计通过率和失败原因 |
| 多数据源查询 | 每个数据源一个子任务 | 统一口径、标记来源 |
| 长报告生成 | 每个章节一个子任务 | 统一风格、补齐引用 |
| 多候选方案 | 每个候选一个子任务 | 评分排序、选择最优 |

业务需要定义:

- 哪些 Skill 可以并发，哪些必须顺序执行。
- 最大并发数、超时、重试和失败策略。
- 子任务失败时是整体失败、部分可用，还是进入人工处理。
- Reducer 如何处理冲突、重复、缺失和低置信结果。
- 并发带来的额外 token、工具调用和外部 API 成本是否可接受。

工程落地:

```text
SkillCall
-> ExecutionPlan
-> 多个 SkillSubCall 并发执行
-> Reducer 汇总
-> 风险复检
-> Review / 结果确认
-> 价值计量和成本归集
```

边界:

- 并发不改变贡献分公式，只缩短执行时间。
- 并发子任务不能绕过授权、源码加密、风险分级、计量和审计。
- 子任务结果不能直接给用户使用，必须经过 Reducer 和最终风险检查。
- 共享写状态、高风险审批、强顺序推理任务不默认启用并发。

### 5.4 数据与工程方案

#### 5.4.1 核心数据对象

| 对象 | 作用 |
|---|---|
| `TaskTypeConfig` | 任务类型、适用部门、默认风险、是否可自动通过 |
| `BaselineWorkloadConfig` | 基准工时、样本来源、可信度、生效版本 |
| `BusinessValueConfig` | 业务价值等级和权重 |
| `TaskComplexityRule` | 复杂度维度和系数映射 |
| `ReviewTemplate` | Review 评分维度、权重、通过阈值 |
| `SkillWeightConfig` | Skill 建议权重、审批权重、生效时间 |
| `ScoreFormulaVersion` | 评分公式版本 |
| `ScoreRecord` | 单次分数记录 |
| `FormulaSnapshot` | 单次分数使用的全部权重和输入 |
| `ScoreAdjustmentLog` | 调整、申诉、复核和审批记录 |
| `SkillSourcePackage` | 加密保存 Skill 源码 / 能力实现包 |
| `ExecutionPlan` | 一次 SkillCall 的并发执行计划 |
| `SkillSubCall` | 并发子任务的执行证据 |
| `AuthorizationGrant` | 谁对哪个能力资产有什么动作权限 |
| `EncryptedSecret` | API Key、license key、结算凭证等敏感凭据引用 |
| `UsageMeterRecord` | 一次调用产生的资源消耗计量 |
| `CostAllocationRecord` | 将消耗归集到成本中心、组织、Skill 或客户 |
| `SettlementAccount` | 内部成本中心、项目、客户或 license 池账户 |
| `RechargeTransaction` | 充值、扣减、退款、调整、冻结流水 |
| `LicenseEntitlement` | 授权主体、资产范围、配额和有效期 |

#### 5.4.2 评分服务

评分服务只做三件事:

1. 读取当时生效的规则。
2. 计算候选分或确认分。
3. 写入不可丢失的公式快照。

伪流程:

```text
任务通过 Review 或自动通过
-> ScoreService 读取任务、Skill、Review、规则版本
-> 计算 saved_hours
-> 读取复杂度、业务价值、Skill 权重、质量系数
-> 执行反刷分检查
-> 应用封顶
-> 生成 ScoreRecord
-> 写 FormulaSnapshot
-> 更新看板聚合
```

#### 5.4.3 规则版本

每次规则变更都生成新版本。

规则版本必须包含:

- 规则类型。
- 规则内容。
- 生效范围。
- 生效时间。
- 审批人。
- 调整原因。
- 影响预览结果。

历史记录读取历史版本，不能被新版本静默覆盖。

### 5.5 页面落地方案

| 页面 | 需要增加或明确的能力 |
|---|---|
| 任务中心 | 展示任务类型、业务价值、风险初判、是否需要 Review |
| Skill 调用页 | 展示候选贡献分和“不会直接入账”的提示，提供 Skill 痛点反馈入口 |
| Review 中心 | 展示 Review 评分模板、维度权重和业务质量标准 |
| 效率看板 | 支持分数解释抽屉、公式快照、申诉入口 |
| Skill 管理 | 展示系统建议权重、业务审批权重、风险、复用指标和员工反馈聚合 |
| 权重计算页 | 解释公式、权重来源、封顶和反刷分 |
| 管理后台 | 配置任务类型、基准工时、业务权重、Review 模板、封顶规则和反馈处理权限 |

### 5.6 试点落地方案

建议先选 1-2 个部门做试点，不要一开始全公司铺开。

试点范围:

- 3-5 个高频任务类型。
- 2-3 个高价值任务类型。
- 1-2 个高风险任务类型。
- 3-5 个核心 Skill。
- 5-10 位 Review 评审人。

试点步骤:

```text
第 1 周: 业务访谈和任务类型梳理
第 2 周: 基准工时、业务价值、Review 模板配置
第 3 周: 小范围真实任务试跑
第 4 周: 复盘分数解释、Review 分歧、封顶和异常
第 5-6 周: 扩大样本并固化第一版规则
```

试点成功标准:

- 员工能理解分数来源。
- 员工能反馈 Skill 使用痛点，但不能管理 Skill。
- Review 评审人能按模板稳定评分。
- 主管认为高价值任务被合理体现。
- 低价值刷分有封顶或异常提示。
- Skill 管理员能基于数据判断优化、推广或下线。

### 5.7 最小可行版本

MVP 不需要一开始实现全部复杂机制，但必须保留核心骨架。

必须做:

- 任务类型配置。
- 基准工时配置。
- 业务价值权重。
- Review 模板。
- Skill 权重配置。
- Skill 使用反馈入口和管理端反馈聚合。
- 公式快照。
- 分数解释。
- 封顶规则。

可以后置:

- 智能异常检测。
- 复杂 ROI 模型。
- 自动化权重调优。
- 跨系统工时采集。
- 多层级业务规则市场。

## 6. 推荐协作流程

### 6.1 初始化规则

```text
确定试点部门
-> 梳理首批任务类型
-> 业务给出基准工时和价值等级
-> 产品建立规则模板
-> 工程配置任务类型、权重和 Review 模板
-> 小范围试运行
-> 根据样本校准
```

### 6.2 规则变更

```text
业务提出规则问题
-> 产品判断是否属于规则问题
-> 数据侧做历史样本影响预览
-> 业务负责人确认
-> 管理员审批生效
-> 工程保存版本和审计
-> 看板展示新旧口径说明
```

### 6.3 周期复盘

建议每月复盘一次，重点看:

- 哪些任务贡献分异常高。
- 哪些低价值任务被高频调用。
- 哪些 Skill 通过率高但业务价值低。
- 哪些 Skill 风险命中率高。
- 哪些 Review 分歧大。
- 哪些基准工时明显偏离真实情况。
- 哪些知识资产被复用，哪些只是沉淀后无人使用。

## 7. 在评分公式中的体现

平台分数应该体现业务思维，但仍保持工程可计算。

```text
确认贡献分 =
节省工时
* 基础点数
* 任务复杂度系数
* 业务价值权重
* Skill 权重
* Review 质量系数
* 复用系数
* 反刷分系数
```

其中:

- 节省工时来自业务基准和实际耗时。
- 任务复杂度来自业务定义的复杂度维度。
- 业务价值权重来自业务价值等级。
- Skill 权重来自系统指标和业务审批。
- Review 质量系数来自业务质量评分。
- 复用系数来自知识资产的真实复用。
- 反刷分系数来自业务对低价值重复行为的识别。

这套公式的重点不是“让分数复杂”，而是让每个分数都能回答:

```text
为什么这个任务有价值？
为什么这个 Skill 值得加权？
为什么这个结果能入账？
为什么这个分数没有被刷出来？
```

## 8. 错误做法

### 8.1 只统计调用次数

问题:

- 会鼓励多调用，而不是做高价值任务。
- 低价值 Skill 容易刷量。
- 管理者无法判断真实业务收益。

### 8.2 只统计节省工时

问题:

- 员工自报不稳定。
- 同样时间差背后的业务价值不同。
- 高风险输出可能节省时间但制造错误。

### 8.3 业务随意人工改分

问题:

- 分数不再可解释。
- 员工无法信任平台。
- 工程快照和审计失去意义。

### 8.4 技术自行定义业务权重

问题:

- 权重看似合理，但脱离真实业务。
- 高风险任务可能被错误降权或放行。
- Skill ROI 会误导管理决策。

## 9. 验收标准

引入业务思维后，平台至少要满足:

- 每个任务类型都有业务定义和适用边界。
- 每个基准工时都有来源、版本和复核机制。
- 每个业务权重都有业务原因和审批记录。
- 每个 Review 模板都能解释业务质量标准。
- 每个 Skill 权重都区分系统建议值和审批生效值。
- 每个分数都能展开公式快照和权重来源。
- 每个封顶或 0 分结果都能说明原因。
- 每次规则变更都能回溯审批人、生效时间和影响范围。
- 历史分数不因新规则生效而被静默覆盖。
- 每次能力调用都能回溯审计、授权、资产版本、风险、价值计量和消耗计量。
- 每笔成本归集都能回到 UsageMeterRecord 和成本公式快照。
- 每次充值、退款、额度调整、license 发放或冻结都能回溯审批和审计。
- 员工端不出现充值、license 发放、成本调整或明文凭据管理入口。
- 员工和外部调用方不能通过调用 Skill 获取 Skill 源码 / 能力实现包。
- 并发 Skill 能回溯 ExecutionPlan、SkillSubCall、Reducer、计量和失败原因。

## 10. 一句话总结

业务思维不是在评分后人工判断“谁该得多少分”，而是在评分前定义清楚什么任务有价值、什么产出可用、什么 Skill 值得推广、什么行为应该被限制。

工程系统的价值，是把这些业务规则稳定地计算出来、解释出来、审计出来。
