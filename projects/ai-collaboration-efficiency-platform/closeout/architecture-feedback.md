## 模板优化候选

- [ ] 原始文档必须先归档，总文档持续演进，分期文档、专题文档和原型说明不能替代总文档。
- [ ] PRD 可以完整表达长期愿景，开发文档必须收敛为可执行范围，并明确本期做什么、不做什么。
- [ ] 开发前必须有冻结清单，至少覆盖页面、接口、数据表、状态机、权限、验收标准。
- [ ] API 草案必须升级为接口契约，至少包含请求参数、响应结构、错误码、权限要求和状态迁移约束。
- [ ] 面向开发 Agent 的开发文档必须显式写出禁止实现项、后续预留项和不得从全文自行扩大范围的规则。

## Prompt / 生成链路优化候选

- [ ] 复杂产品文档生成前，应先区分四类文档关系：原始归档、总文档、分期文档、专题补充文档。
- [ ] 当用户追加阶段规划、商业化规划或技术边界时，生成链路应先更新总文档，再按需要同步分期执行文档。
- [ ] 遇到“方案完整但研发不可执行”的反馈时，应优先补冻结清单、接口契约、数据库落地细节、状态机和验收标准，而不是继续扩展愿景。
- [ ] 对二期或后续能力，应使用明确措辞：不进入本期 Sprint、不创建生产表、不开放接口、不纳入验收。
- [ ] 产品原型反馈如果改变业务逻辑、权限边界或计算机制，应同步更新 PRD 和开发文档，而不只改原型说明。

## 工作流优化候选

- [ ] 项目经验先进入项目内 `closeout/architecture-feedback.md`，经用户审核后再决定是否转入 `docs/architecture-inbox/`。
- [ ] 不从项目反馈直接修改长期记忆、stable skill、harness、workflow、registry 或 governance。
- [ ] 架构沉淀应拆成三层：项目证据、通用建议、待用户拍板动作。
- [ ] 若要把项目经验转成稳定规则，应单独创建受监督变更方案，并说明优势、风险、替代方案、执行范围和验证方法。

## 数据模型候选

- [ ] 数据模型必须落到数据库工程细节，不只列字段。
- [ ] 数据库设计应至少包含主外键、唯一约束、索引、软删除策略、审计字段规范、枚举表或代码枚举策略、JSONB 字段结构示例。
- [ ] 字段治理应包含角色可见性和权限边界，区分使用者、管理者、评审者、HR、外部客户和系统运行时。
- [ ] 涉及看板指标时，字段应能追溯到事实表、公式快照、聚合口径和权限过滤规则。

## 架构决策候选

- [ ] 涉及评分、价值、ROI、风险的产品，必须列变量来源、业务 owner、采集方式、默认值、更新频率和公式快照。
- [ ] 使用者、管理者、评审者、观察者必须分权，避免被计算的人同时修改计算规则。
- [ ] 看板指标必须具备证据链：数据源、计算公式、快照、权限、明细追溯路径。
- [ ] 能力资产交付必须区分使用权、管理权、源码权、导出权；调用授权不等于源码访问授权。
- [ ] 涉及 Skill、Prompt、Workflow、工具代码或私有规则的产品，应把加密、授权、版本和审计作为同一组控制面能力设计。
- [ ] AI 能力优化应优先走“反馈 -> 测试集 -> 候选版本 -> 对比 -> 多人评审 -> 发布”闭环，不能让反馈直接自动上线。
- [ ] 商业化能力应晚于审计、授权、版本、风险、价值计量和成本归集；没有可审计计量证据时，不应先做充值、license 或真实结算。
