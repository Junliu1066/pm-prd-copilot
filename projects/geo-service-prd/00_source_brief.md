## 1. 使用方式

本文件把原始前置材料归一化为 PRD 写作输入。原始材料中的 19 个系统不是 PRD 正文结构，而是用于帮助确定产品判断、业务目标、用户对象、场景优先级、MVP 边界、交付机制、商业模式和风险边界。

PRD 应按既有 PRD 框架输出，不逐字复述前置材料，不把前置材料的章节直接当成 PRD 目录。

---

## 2. 主线判断

当前主线不是“做一个 GEO SaaS 平台”，而是：

> 把 GEO 服务包装成一套可销售、可交付、可复购、可逐步 Agent 化放大的服务产品。

第一阶段的产品形态应是：

> GEO 免费检测 + GEO 体检报告 + GEO 深度诊断报告 + 后续优化 / 代运营转化。

内部 Agent 和工作台是交付效率工具，不是 MVP 对客户销售的核心产品。

## 3. 已确认事实

| 编号 | 事实 |
|---|---|
| F1 | 用户希望把“给别的公司提供 GEO 服务赚钱”拆成可落地生意。 |
| F2 | 服务不是卖 GEO 技术概念，而是帮企业抢占 AI 搜索时代的推荐入口。 |
| F3 | 客户核心关心 AI 是否推荐自己、是否推荐竞品、是否正确描述品牌。 |
| F4 | 第一阶段不要先做复杂平台，先做 GEO 诊断报告服务。 |
| F5 | 推荐商业路径为免费检测获客、报告成交、优化项目赚钱、月度代运营复购、Agent 降本。 |
| F6 | 适合客户优先是高客单、高信任、强比较、强决策链路行业。 |
| F7 | GEO 交付闭环应包含检测、诊断、策略、优化、发布、监控、复盘、续费。 |
| F8 | 风险边界明确：不能承诺一定被 AI 推荐，不能做虚假评价、虚假报道、伪造背书或黑帽 GEO。 |
| F9 | 用户补充：市面同行常见路径是 Web 端引流到私域，在私域售卖服务，并在服务期间给用户可溯源的数据看板实时反馈。 |

## 4. 当前假设

| 编号 | 假设 | 影响 |
|---|---|---|
| A1 | 首期目标客户优先选 ToB SaaS、企业服务、财税 / 法律 / 咨询、教育 / 医美等高客单行业。 | 影响销售样板、问题库和报告案例。 |
| A2 | MVP 用半自动交付即可验证商业价值，不需要完整系统。 | 影响研发和工具投入边界。 |
| A3 | 首期交付物以报告、检测表、问题库、优化计划和报价方案为主。 | 影响 PRD 范围。 |
| A4 | MVP 需要轻量客户侧数据看板，但不等于完整 SaaS 后台；看板重点是可溯源、可反馈、可促进信任和续费。 | 影响原型图层、交付物范围和数据结构。 |
| A5 | Agent 用于内部生成问题库、分析缺口、生成报告初稿和检查风险。 | 影响 AI 能力边界。 |

## 5. 开放问题

| 优先级 | 问题 | 建议 |
|---|---|---|
| P0 | 第一批试点行业是哪 3 个？ | 建议先选 ToB SaaS、财税 / 法律 / 咨询、教育 / 医美高客单服务。 |
| P0 | 免费检测和体检报告的真实交付深度如何区分？ | 免费检测只做 3 个问题和初步建议；体检报告做 30 个问题、3 平台、3 竞品。 |
| P1 | 检测平台首期覆盖哪些？ | 先做人工 / 半自动记录模板，平台范围按客户行业选择。 |
| P1 | 报告是否需要视觉模板？ | MVP 先标准化内容结构，视觉模板作为销售资产后置。 |
| P1 | 数据看板首期做到什么深度？ | 建议 MVP 做客户可见的项目看板：检测问题、原始答案、来源、指标趋势、优化动作和更新时间；复杂权限、自动检测和代理商后台后置。 |
| P1 | 月度代运营是否纳入 MVP？ | MVP 设计转化路径、看板反馈和月报模板，真正规模化代运营作为 V1 / V2。 |
| P2 | 是否做完整客户后台或代理商后台？ | 放到 V3；MVP 只做轻量可溯源数据看板。 |

## 6. 对 PRD 的生成要求

PRD 需要回答这些问题：

- 这套 GEO 服务产品解决谁的什么问题？
- 第一阶段卖什么，不卖什么？
- 免费检测、体检报告、深度诊断和代运营如何衔接？
- 每个产品包的交付物、验收标准和价格区间是什么？
- Web 引流、私域承接、服务售卖和数据看板如何串成转化链路？
- 内部交付流程如何保证报告质量和合规边界？
- Agent 在哪个环节提高效率，哪些结论必须人工复核？
- 如何通过指标判断这个服务产品是否成立？

## 7. 不应写成的方向

| 错误方向 | 原因 |
|---|---|
| 直接写成 GEO SaaS 工作台 PRD | 原始材料明确第一阶段不要先做平台。 |
| 把 19 个系统直接当 PRD 目录 | 这些是前置思考，不是交付文档框架。 |
| 把 Agent 能力包装成客户可见主功能 | 当前 Agent 应先作为内部降本工具。 |
| 过多写页面和后台权限 | MVP 核心是服务产品和交付物，不是软件系统。 |
| 承诺 AI 推荐结果 | 违反合规和事实边界。 |
