## 5. 事件状态流转图

适合放在 PRD「状态流转」中，用于把事件、AI 摘要、审核和发布状态画成研发可执行的状态机。

```mermaid
stateDiagram-v2
  [*] --> Collected
  state "采集到事件" as Collected
  state "待清洗" as Cleaning
  state "清洗失败" as CleanFailed
  state "待 AI 解读" as AiPending
  state "待审核" as ReviewPending
  state "已发布" as Published
  state "已更新" as Updated
  state "已拦截" as Blocked
  state "已过期" as Expired
  state "已撤回" as Revoked

  Collected --> Cleaning: 原始资讯入库
  Cleaning --> CleanFailed: 来源不可读 / 清洗异常
  Cleaning --> AiPending: 清洗成功并关联实体
  AiPending --> ReviewPending: 摘要生成成功或命中复核条件
  AiPending --> ReviewPending: 来源不足 / 高影响事件
  ReviewPending --> Published: 审核通过
  ReviewPending --> Blocked: 命中 P0 或审核拒绝
  Published --> Updated: 新来源 / 新行情 / 用户反馈
  Updated --> ReviewPending: 重新检查来源和风险
  Published --> Expired: 事件时效结束
  Published --> Revoked: 来源撤回 / 合规要求
  Blocked --> ReviewPending: 申诉或规则复核
```

看图要点：

- 这张图比文字状态表更适合研发实现状态机。
- 测试时要覆盖：低置信、命中 P0、人工通过、发布后更新、撤回、过期。

## 7. AI 模型路由决策树

适合放在 PRD「AI 模型选型」中，用于把模型选择从结论变成可执行规则。

```mermaid
flowchart TD
  T["任务进入 AI 路由"] --> Q1{"任务类型"}

  Q1 -->|清洗 / 分类 / 实体抽取| Q2{"是否需要实时性"}
  Q2 -->|高实时| FastSync["快速低成本模型，同步或准实时"]
  Q2 -->|可异步| FastAsync["快速低成本模型，异步批处理"]

  Q1 -->|事件摘要 / 概念解释| Q3{"是否需要长上下文"}
  Q3 -->|否| Mid["中等能力模型 + RAG"]
  Q3 -->|是| Strong["高能力长上下文模型 + RAG"]

  Q1 -->|个股深度分析 / 长文综合| Strong
  Q1 -->|内容安全 / 合规拦截| Safety["规则引擎 + 分类模型"]

  FastSync --> Q4{"是否高风险"}
  FastAsync --> Q4
  Mid --> Cite{"关键结论是否有来源"}
  Strong --> Cite
  Safety --> Q4

  Q4 -->|是| Human["人工复核 / 审核队列"]
  Q4 -->|否| Q5{"关键结论是否有来源"}
  Q5 -->|否| Insufficient["输出不足以判断 / 待核验"]
  Q5 -->|是| Publish["发布或展示 AI Insight"]
  Cite -->|否| Insufficient["输出不足以判断 / 待核验"]
  Cite -->|是| Q4

  Human --> H1{"审核结果"}
  H1 -->|通过| Publish
  H1 -->|修改后通过| Publish
  H1 -->|拒绝| Block["拦截 / 隐藏 / 降权"]

  Publish --> Log["记录模型、Prompt、来源、时间、置信度"]
  Block --> Log
  Insufficient --> Log
```

看图要点：

- 这张图避免“模型选型像结论”，让选型变成可执行的路由规则。
- 研发实现时可以把每个判断点变成 router 规则、审核规则或日志字段。

## 9. 风险控制闭环图

适合放在 PRD「合规与风险控制」前面，用于强化金融投研产品的安全边界。

```mermaid
flowchart LR
  Source["风险来源"] --> Detect["风险识别"]

  Mark --> Review["人工审核"]

  Limit --> Log["日志追踪"]
  Log --> Improve["复盘优化"]
  Improve --> Detect

  Source --> S1["无来源重大事实"]
  Source --> S2["传闻包装成确定事实"]
  Source --> S3["买卖建议 / 收益承诺"]
  Source --> S4["未授权研报正文"]
  Source --> S5["AI 强推断"]

  Detect --> D1["规则检查"]
  Detect --> D2["引用一致性检查"]
  Detect --> D3["置信度检查"]

  Limit --> L3["风险提示展示"]
  Limit --> L4["撤回"]
```

看图要点：

- 这张图说明：风险不是只在发布前检查一次，而是识别、审核、发布限制、日志追踪、复盘优化的闭环。
- 对金融投研产品尤其关键，能避免读者忽略合规边界。
