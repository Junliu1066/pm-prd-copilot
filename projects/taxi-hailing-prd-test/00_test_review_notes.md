## 3. 初步判断

章节结构本身没有改坏，已经按用户要求生效。

真正暴露的问题是生成链路质量：

1. 需求解析会把“测试目的 / 文档目的”当成产品业务目标。
2. rule mode 的业务识别太粗，不能生成真实行业 PRD。
3. 原型章节仍是旧机制，没有切换到页面说明和跳转关系。
4. 通用 fallback 内容带有历史场景残留，例如导出、Excel、异步导出。
5. AI 模型选型需要按项目是否涉及 AI 判断，而不是默认输出一段。

---

## 4. 建议下一步

不要继续直接改稳定链路。

建议用户先审核：

1. 第 14、15 章后置后的阅读体验是否保留。
2. 是否批准把“页面说明 + 页面跳转关系”正式并入长期模板和生成器。
3. 是否需要重做 rule mode 的业务识别逻辑，避免泛化内容污染真实 PRD。
4. 是否需要把测试输入中的“测试目的”与“产品需求”分层，避免被解析器混用。
