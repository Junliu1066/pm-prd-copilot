# 临时需求交付包生成器 - Requirement Brief

## 基本信息

- Project ID: temp-generated-project
- 类型: temporary_project / internal_tool
- 紧急度: P2
- 目标上线时间: 未指定
- 文档状态: Draft
- 最后更新: 2026-04-30

## 问题定义

当前临时产品想法或客户需求进入产品经理工作区时，常见问题是输入材料不完整、PRD 和开发文档边界混杂、原型说明缺失、交付检查不一致。临时项目尤其容易把工作产物写散到稳定目录或根目录，增加后续清理和治理风险。

本项目要提供一个轻量的临时交付包生成器，用于把一段业务输入快速整理成:

- 需求澄清与假设
- PRD
- 低保真原型说明
- 开发文档
- 交付检查和 closeout

## 目标用户

| 用户 | 目标 | 痛点 |
|---|---|---|
| 产品经理 | 快速把临时想法变成可评审材料 | 输入零散、边界不清、容易漏验收 |
| 创业团队负责人 | 快速判断一个想法是否值得开发 | 缺少统一交付包，沟通成本高 |
| 研发负责人 | 依据稳定文档评估开发量 | PRD 不够工程化，验收和风险不清 |
| AI 协作执行者 | 在隔离目录内生成完整产物 | 容易误改稳定架构或长期规则 |

## 业务目标

- 在 30 分钟内从临时输入形成可读的第一版交付包。
- 降低临时项目污染稳定治理架构的概率。
- 让后续评审者能快速判断项目是否继续推进、补充信息或关闭。

## 核心场景

1. 用户给出一句话想法，系统生成需求澄清和待确认问题。
2. 产品经理选择 MVP 范围，系统生成 PRD 和验收标准。
3. 团队需要原型沟通，系统生成低保真页面说明和静态预览。
4. 研发准备估时，系统生成开发文档、任务拆分和检查项。
5. 临时项目结束后，系统生成 closeout，只记录项目内反馈，不修改稳定架构。

## 已确认事实

- 本次项目是临时生成使用。
- 产物必须全部写入 `projects/temp-generated-project/`。
- 不允许修改稳定治理架构和根目录文件。
- 需要中文交付报告。

## 工作假设

- 临时项目默认不接入真实账号系统、支付系统和外部模型调用。
- MVP 使用人工输入 + 规则化模板 + 可选 AI 生成的方式，不要求自动化 pipeline。
- 静态 HTML 原型足够支撑临时评审。
- 交付包的质量优先于功能复杂度。

## Open Questions

### P0: 影响方向

- 该工具最终是只供内部产品经理使用，还是要开放给外部客户或外包团队？
- 是否需要真正接入 AI 模型自动生成，还是保持为文档化流程样例？
- 是否需要用户登录和权限隔离？

### P1: 可先假设

- 输入材料支持纯文本、Markdown、图片 OCR 还是表格？
- 交付包是否需要导出 zip？
- 是否需要保存多个版本并做差异对比？
- 是否需要接入现有 harness 做自动检查？

### P2: 后续细节

- 是否需要团队协作评论？
- 是否需要任务状态看板？
- 是否需要生成外部分发版短文件名包？

## 建议范围

### MVP

- 单项目输入区。
- 自动生成需求澄清、PRD、原型说明、开发文档四类草稿。
- 展示待确认问题和风险。
- 提供交付完整性检查清单。
- 所有产物限定在项目目录内。

### V1

- 支持版本历史和差异对比。
- 支持一键导出项目 zip。
- 支持人工标记问题优先级和决策状态。
- 支持基于模板生成外部分发包。

### Later

- 接入模型生成和引用检查。
- 接入自动 harness 检查。
- 支持多角色协作和评论。
- 支持跨项目复用，但必须经过稳定变更审批。
