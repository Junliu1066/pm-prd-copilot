## 1. 一句话摘要

临时需求交付包生成器是一个面向产品经理和小团队的内部工具，用于把零散业务输入快速整理成需求澄清、PRD、原型说明、开发文档和交付检查，帮助临时项目在独立目录内完成可评审交付。

## 2. 背景与问题

临时项目通常有三个典型问题:

- 输入不完整: 只有一句话想法或聊天记录，缺少用户、范围、验收和风险。
- 产物不成套: 有 PRD 但没有原型，有原型但没有开发拆分，有开发说明但没有验收标准。
- 写入边界不稳: 临时探索容易误改根目录、稳定 workflow、harness 或 governance 文件。

本项目用轻量交付包方式解决上述问题，不改变现有稳定架构。

## 3. 目标与非目标

### 3.1 产品目标

- 帮用户在一个页面内完成临时需求输入、澄清和产物生成。
- 让 PRD、原型说明、开发文档和检查报告保持同一项目目录。
- 在生成结果中明确事实、假设、阻塞问题和非阻塞问题。
- 在项目 closeout 中记录改进建议，而不是直接修改长期规则。

### 3.2 用户目标

- 产品经理能快速得到可评审初稿。
- 研发能看到明确实现边界和验收标准。
- 项目负责人能判断是否继续推进。
- 后续执行者能知道哪些问题待确认。

### 3.3 非目标

MVP 不做:

- 不修改产品经理 skill、harness、workflow 或 governance。
- 不接入生产账号体系。
- 不做团队权限、审批流和多人实时协作。
- 不承诺 AI 自动生成内容完全正确。
- 不把临时偏好写成长期偏好。

## 4. 用户与使用场景

| 场景 | 触发 | 用户动作 | 成功结果 |
|---|---|---|---|
| 临时想法成稿 | 只有一句话或聊天需求 | 粘贴输入并选择产物类型 | 生成需求简报和 PRD 初稿 |
| 研发评估 | PRD 初稿需要估时 | 查看开发文档和任务拆分 | 能判断工作量和依赖 |
| 原型沟通 | 需求需要页面化说明 | 打开原型说明或 HTML 原型 | 对页面结构形成共识 |
| 项目关闭 | 临时探索结束 | 生成 closeout | 记录结果和架构反馈 |

## 5. MVP 范围

### 5.1 输入管理

功能:

- 支持粘贴文本输入。
- 支持填写项目英文名、中文名、目标用户、业务目标。
- 支持选择产物类型: PRD、原型说明、开发文档、检查报告。
- 支持标记项目性质: 临时 / 内部 / 外部分发。

验收:

- 必填字段为空时给出错误提示。
- 项目英文名只允许小写字母、数字和短横线。
- 明确展示写入目录。

### 5.2 需求澄清

功能:

- 自动整理事实、假设、P0/P1/P2 问题。
- 输出建议 MVP / V1 / Later 范围。
- 对阻塞问题做显著标记。

验收:

- 每个 P0 问题都说明为什么阻塞。
- 每个假设都能被后续 PRD 引用。
- 未提供的信息不得伪装成事实。

### 5.3 PRD 生成

功能:

- 生成背景、目标、用户、范围、流程、功能需求、非功能需求、指标、风险。
- 保留非目标和边界说明。
- 生成验收标准。

验收:

- PRD 内至少包含目标、非目标、MVP 范围和验收标准。
- 风险项包含影响、触发条件和缓解方案。
- 与需求简报中的假设保持一致。

### 5.4 原型说明与静态预览

功能:

- 生成页面信息架构。
- 生成关键页面线框说明。
- 提供一个静态 HTML 低保真原型。

验收:

- 至少覆盖输入工作台、生成结果、检查面板三个页面区块。
- 原型能直接在浏览器打开。
- 原型不依赖外部网络资源。

### 5.5 开发交付文档

功能:

- 给出技术栈建议、模块边界、数据结构、接口草案、任务拆分、测试策略。
- 明确不修改稳定架构。
- 明确交付检查步骤。

验收:

- 开发文档能支持研发做第一轮估时。
- 每个阶段都有交付物和验收标准。
- 写入边界清晰，不要求改稳定目录。

## 6. 核心流程

```mermaid
flowchart LR
  A["输入临时需求"] --> B["拆分事实和假设"]
  B --> C["生成待确认问题"]
  C --> D["选择 MVP 范围"]
  D --> E["生成 PRD"]
  E --> F["生成原型说明"]
  F --> G["生成开发文档"]
  G --> H["运行交付检查"]
  H --> I["写入项目 closeout"]
```

## 7. 信息架构

| 一级模块 | 二级内容 | 说明 |
|---|---|---|
| 项目输入 | 基本信息、业务输入、约束 | 形成原始输入和需求简报 |
| 澄清面板 | 事实、假设、问题分级 | 帮用户补齐方向性信息 |
| 产物生成 | PRD、原型说明、开发文档 | 分文件输出 |
| 检查中心 | 写入范围、完整性、一致性 | 防止遗漏和越界 |
| Closeout | 结果、风险、架构反馈 | 项目内收口 |

## 8. 功能需求

### FR-1 项目创建

- 用户可以输入项目英文名。
- 系统展示目标目录。
- 系统阻止写入禁止路径。

### FR-2 输入解析

- 系统将输入分为事实、假设、问题。
- 系统根据影响范围把问题分为 P0、P1、P2。
- 系统对 P0 问题提示“需要确认后再进入开发”。

### FR-3 文档生成

- 系统生成需求简报、PRD、原型说明和开发文档。
- 每个文件带状态、更新时间和关联文档。
- 系统保留“本次为临时生成”的说明。

### FR-4 原型预览

- 系统提供静态 HTML 低保真原型。
- 原型展示输入区、澄清结果、产物列表和检查状态。
- 原型不保存真实数据。

### FR-5 交付检查

- 系统检查产物是否都在项目目录。
- 系统检查 PRD、原型说明和开发文档是否相互引用。
- 系统输出 round 1 和 round 2 检查结果。

## 9. 非功能需求

| 分类 | 要求 |
|---|---|
| 安全 | 不写入项目目录外文件，不调用生产凭证 |
| 性能 | 纯本地文档生成场景下 5 秒内完成静态页面打开 |
| 可维护性 | 文档命名清晰，后续可直接删除项目目录 |
| 可审计 | closeout 记录假设、检查结果和未决问题 |
| 可移植 | 静态 HTML 不依赖外部资源 |

## 10. 成功指标

| 指标 | MVP 目标 | 说明 |
|---|---:|---|
| 交付包完整率 | 100% | 需求简报、PRD、原型说明、开发文档、检查报告齐全 |
| 写入边界合规率 | 100% | 不改禁止路径和根目录文件 |
| P0 问题显式化 | 100% | 阻塞问题必须列出 |
| 原型可打开率 | 100% | `prototype/index.html` 可直接打开 |
| 人工复查时间 | <= 15 分钟 | 读者能快速判断是否继续推进 |

## 11. 风险与缓解

| 风险 | 影响 | 缓解 |
|---|---|---|
| 输入过少导致 PRD 假设过多 | 后续开发方向可能偏离 | 在需求简报中标记 P0/P1/P2 问题 |
| 临时需求被误当成稳定规则 | 污染治理架构 | 所有改进只写入 closeout |
| 生成文档互相矛盾 | 评审成本上升 | 交付前做一致性检查 |
| 原型被误认为最终 UI | 设计预期偏差 | 明确标注低保真、仅用于结构讨论 |

## 12. 上线与验收

MVP 验收条件:

- `projects/temp-generated-project/` 目录存在。
- 项目目录内包含需求简报、PRD、原型说明、开发文档、交付检查和 closeout。
- `prototype/index.html` 可直接打开。
- 检查报告确认未修改稳定目录。
- closeout 记录是否有架构反馈。
