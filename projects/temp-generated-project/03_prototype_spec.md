## 1. 原型目标

该原型用于说明临时需求交付包生成器的主路径和页面布局，不代表最终视觉设计。核心目标是让评审者看懂:

- 临时需求如何输入。
- 系统如何拆出事实、假设和待确认问题。
- 产物如何按 PRD、原型说明、开发文档和检查报告组织。
- 写入边界和 closeout 如何展示。

## 2. 页面结构

```text
┌──────────────────────────────────────────────────────────────┐
│ 顶部栏: 项目名 / 目录边界 / 当前状态                          │
├──────────────────────┬───────────────────────────────────────┤
│ 左侧: 输入与约束      │ 右侧: 生成结果概览                     │
│ - 项目英文名          │ - 需求简报                             │
│ - 业务输入            │ - PRD                                  │
│ - 产物选择            │ - 原型说明                             │
│ - 禁止路径提醒        │ - 开发文档                             │
├──────────────────────┴───────────────────────────────────────┤
│ 中部: 澄清与范围                                               │
│ - Facts / Assumptions / P0 / P1 / P2                           │
├──────────────────────────────────────────────────────────────┤
│ 底部: 交付检查与 closeout                                      │
│ - 写入范围检查 / 完整性检查 / 一致性检查 / 架构反馈             │
└──────────────────────────────────────────────────────────────┘
```

## 3. 关键区块

### 3.1 顶部状态栏

展示:

- 产品名: 临时需求交付包生成器
- 项目目录: `projects/temp-generated-project/`
- 状态: 临时生成 / 不进入稳定架构

交互:

- 点击“打开原型”在本地浏览器查看静态页面。
- 点击“查看检查”跳转到底部检查区。

### 3.2 输入区

字段:

- 项目英文名
- 项目中文名
- 一句话需求
- 目标用户
- 业务目标
- 写入边界

状态:

- 空状态: 提示输入一句话业务需求。
- 错误状态: 项目英文名不合法、缺少目标用户、写入目录不在 `projects/` 下。
- 已完成状态: 显示“可生成需求简报”。

### 3.3 产物概览区

展示四类产物:

- 需求简报
- PRD
- 原型说明
- 开发文档

每个产物展示:

- 生成状态
- 最后更新时间
- 是否需要人工确认
- 关联风险数量

### 3.4 澄清与范围区

分栏展示:

- Facts: 已确认事实。
- Assumptions: 工作假设。
- P0: 会影响方向的问题。
- P1: 可先假设、评审时调整的问题。
- P2: 细节问题。

### 3.5 检查与 closeout 区

检查项:

- 文件是否只在项目目录内。
- 是否覆盖 PRD、原型说明、开发文档。
- 是否记录 round 1 和 round 2 检查。
- 是否把架构优化建议写入 closeout。

## 5. 原型验收

- 页面可离线打开。
- 页面宽度在桌面和窄屏下不出现明显遮挡。
- 页面不依赖外部脚本、图片或字体。
- 页面显式展示“临时生成”和“只写项目目录”。
- 页面不包含稳定治理架构修改入口。
