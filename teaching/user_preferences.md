# User Preferences

- Prefer PRD systems that learn structure and judgment from examples without hard-coding a single PDF template.
- Prefer supervised learning: proposals first, user approval before memory, skill, or harness updates.
- Prefer broad user-pool analysis before target-user narrowing.
- Prefer scenario selection based on minimum cost and maximum return.
- Prefer interactive HTML prototypes delivered as cross-platform zip packages with editable `html/css/js` source, plus a `standalone.html` fallback.
- Prefer PM-editable prototypes: ordinary text should be editable by double-clicking in the browser.
- Prefer prototype actions that represent real workflows to navigate to destination pages with carried context, instead of placeholder modals.
- Prefer multiple related prototype screenshots to be consolidated into one navigable multi-page HTML prototype.
- Prefer PRDs to include a project-fit visual reading layer: page descriptions, page jump relationships, and a prototype layer are the baseline; mind maps, swimlanes, MVP maps, risk loops, status flows, permission matrices, and story maps should be selected by project need and placed in the relevant sections.
- Interpret “我自己开发 / 自己开发” as user-led implementation, not personal-only product usage; confirm product audience separately when unclear.
- Prefer explicit use of the user's own operating frameworks when approved: multi-manager/steward systems, Skill/MCP routing, harness, audits, and learning mechanisms can be visible product and governance architecture rather than hidden internals.
- Prefer audience-layered documents: PRD, engineering development document, and Codex internal execution document should be distinct artifacts with different responsibilities.
- Prefer Codex internal development documents to be paired with development-ready PRDs, Codex development planning, or semi-automated delivery requests, and to be directly executable with task packages, write boundaries, validation commands, human gates, and minimal-fix strategies.
- Prefer development-ready zip packages to be complete for the intended audience, while keeping internal and external/protected packages separated by redaction and distribution policy; use an English-path compatibility package when needed.
- When told not to modify yet, report only; do not write memory, project files, proposals, packages, or stable instructions.
